# SafeMetrica Hazard Taxonomy & Company Risk Discovery Engine v1

- 기준일: 2026-06-13
- 문서 성격: 위험요인 마스터, 태깅, 사업장별 후보 발견 및 운영기록 연결 기준
- 구현 범위: 문서화만 포함하며 애플리케이션 코드, 데이터베이스, API 변경은 포함하지 않음

## 1. 목적

이 문서는 SafeMetrica의 위험요인 마스터와 업종·설비·물질·환경·작업유형 태깅 구조를 정의하고, 이를 현장 데이터 및 근로자 참여 기록과 결합하여 사업장별 위험요인 후보를 발견·검토·축적하는 v1 기준을 정한다.

SafeMetrica는 위험성평가를 대신 수행하는 서비스가 아니다. SafeMetrica는 위험성평가 이후의 공유·확인·참여·검토·증빙 흐름을 운영기록으로 남기는 SaaS다. 이 문서의 분류, 후보, 점수 및 상태는 위험성평가 결과나 법률상 판단을 자동으로 확정하지 않는다.

## 2. SafeMetrica 위험요인 발견 엔진의 정의

SafeMetrica 위험요인 발견 엔진은 KRAS의 6대 유해·위험요인 분류와 업종·설비·물질·작업환경·근로자 참여 데이터를 기반으로 사업장별 위험요인 후보를 구성하는 규칙·데이터 연결 계층이다.

엔진의 역할은 다음과 같다.

1. 공통 `hazard_master`에서 사업장 특성과 관련된 위험요인을 검색한다.
2. 업종, 보유 설비, 취급 물질, 작업환경 및 작업유형 태그의 일치도를 계산한다.
3. TBM, 현장 제보, 아차사고, 개선제안 등 근로자 참여 신호를 후보의 근거로 연결한다.
4. 후보의 출처, 생성 이유, 중복 관계, 검토 이력을 추적 가능한 형태로 남긴다.
5. 관리자에게 검토 대상을 제시하고, 승인된 항목만 회사별 위험요인 원장으로 연결한다.

엔진이 생성한 결과는 어디까지나 검토 후보이며, 현장 조사, 관계자 의견 및 사업장 조건을 확인하지 않은 채 위험성평가 항목이나 조치로 확정해서는 안 된다.

## 3. KRAS 6대 유해·위험요인 분류 참고 기준

`hazard_master`의 최상위 참고 분류는 KRAS에서 제시하는 다음 6대 유해·위험요인 분류를 사용한다.

| 코드 후보 | 분류 | SafeMetrica 활용 예시 |
| --- | --- | --- |
| `mechanical` | 기계적 요인 | 끼임, 절단, 충돌, 낙하·비래 등 후보 분류 |
| `electrical` | 전기적 요인 | 감전, 아크, 정전기, 전기화재 등 후보 분류 |
| `chemical` | 화학적 요인 | 가스, 증기, 분진, 유해물질 노출 등 후보 분류 |
| `biological` | 생물학적 요인 | 세균, 바이러스, 곰팡이, 동식물 매개 노출 등 후보 분류 |
| `work_environment` | 작업특성 요인 | 고소·중량물·반복·야간 작업 등 작업 특성 후보 분류 |
| `work_condition` | 작업환경 요인 | 소음, 진동, 고온·저온, 조도, 공간 조건 등 후보 분류 |

이 분류는 후보를 일관되게 탐색하고 설명하기 위한 참고 체계다. 실제 사업장의 위험요인은 복수 분류에 걸칠 수 있으므로 하나의 상위 분류만으로 현장 판단을 제한하지 않는다. 세부 명칭과 매핑은 향후 공신력 있는 최신 원문을 확인하고 사람의 검수를 거쳐 버전 관리한다.

## 4. 공식 위험요인 수 오인 표현 금지

SafeMetrica의 제품 설명, 화면, 영업자료, 보고서 및 고객용 Export에서 특정 숫자를 KRAS의 공식 위험요인 총수로 단정해서는 안 된다. 특히 아래 두 요소를 결합해 공식 표준처럼 설명하는 문구를 금지한다.

- 숫자: `673개`
- 대상: `KRAS 위험요인`

위 숫자는 제품의 위험요인 마스터 규모, 공인 목록의 총수 또는 위험성평가 충족 범위를 나타내는 마케팅 문구로도 사용하지 않는다.

## 5. 673개 수치 정정 기준

673개는 KRAS 공식 위험요인 수로 확인된 기준이 아니며, 다른 법령·기준 수치와 혼동된 표현일 가능성이 있으므로 SafeMetrica 제품 설명에는 사용하지 않는다.

기존 문서나 화면에서 해당 수치가 발견되면 다음 절차를 따른다.

1. 해당 문구의 출처와 작성 맥락을 기록한다.
2. 공신력 있는 원문에서 수치의 대상과 기준일을 재확인한다.
3. 확인 전에는 숫자를 삭제하고 “KRAS 6대 분류를 참고한 위험요인 마스터”로 정정한다.
4. 특정 목록의 건수를 표시해야 한다면 SafeMetrica 내부 데이터 버전의 실제 활성 항목 수로 명명하고, 공식 기준 수와 분리한다.
5. 정정 이력에는 변경 전 의미, 변경 사유, 검토자 및 적용 문서 범위를 남긴다.

## 6. `hazard_master` 논리 구조

`hazard_master`는 공통 위험요인 후보의 기준 카탈로그다. 실제 물리 테이블을 확정하는 문서가 아니며, v1 논리 필드는 다음과 같다.

| 논리 필드 | 설명 |
| --- | --- |
| `hazard_id` | 안정적인 내부 식별자 |
| `hazard_code` | 중복되지 않는 관리 코드 |
| `name` | 위험요인 표준 표시명 |
| `description` | 위험 발생 조건과 범위에 대한 중립적 설명 |
| `kras_category` | 6대 참고 분류 코드 |
| `accident_type_candidates` | 관련될 수 있는 사고 유형 후보 |
| `health_effect_candidates` | 관련될 수 있는 건강 영향 후보 |
| `source_refs` | 출처명, 문서 버전, 확인일 등 근거 메타데이터 |
| `industry_tags` | 관련 업종 태그 목록 |
| `equipment_tags` | 관련 설비 태그 목록 |
| `material_tags` | 관련 물질 태그 목록 |
| `environment_tags` | 관련 작업환경 태그 목록 |
| `work_type_tags` | 관련 작업유형 태그 목록 |
| `synonyms` | 검색 및 중복 판별용 동의어 |
| `exclusion_rules` | 명백한 오탐을 줄이는 제외 조건 후보 |
| `review_status` | 초안, 검토중, 승인, 비활성 등 마스터 검수 상태 |
| `version` | 분류·내용 변경 추적 버전 |
| `effective_from`, `retired_at` | 적용 및 비활성 시점 |

운영 원칙은 다음과 같다.

- 원문 출처와 SafeMetrica가 작성한 설명을 구분한다.
- 하나의 위험요인은 여러 태그와 연결할 수 있다.
- 동의어는 검색을 돕지만 별도 위험요인을 임의로 합치기 위한 근거로만 사용하지 않는다.
- 삭제보다 비활성화와 버전 이력을 우선하여 기존 운영기록의 해석 가능성을 보존한다.

## 7. `industry_tags`

`industry_tags`는 사업장의 업종 및 세부 공정 특성과 위험요인을 연결한다.

| 논리 필드 | 예시 역할 |
| --- | --- |
| `industry_code` | 표준 또는 내부 업종 코드 |
| `industry_name` | 업종 표시명 |
| `process_scope` | 업종 내 적용 공정 범위 |
| `match_weight` | 후보 검색 시 상대 가중치 |
| `source_basis` | 태깅 근거와 검토 정보 |

사업자등록 업종명만으로 후보를 확정하지 않는다. 같은 업종에서도 실제 공정, 설비 및 작업 방식이 다르므로 업종 태그는 초기 검색 범위를 좁히는 신호로만 사용한다.

## 8. `equipment_tags`

`equipment_tags`는 사업장이 보유하거나 사용하는 기계·기구·차량·전기설비·가설설비와 위험요인을 연결한다.

- 설비 종류, 모델군, 동력원, 방호장치, 사용 위치 및 사용 상태를 분리할 수 있어야 한다.
- 단순 보유와 실제 가동을 구분한다.
- 점검, 청소, 정비, 고장 처리 등 비정상 작업 시나리오를 별도 작업유형과 결합한다.
- 설비명이 유사하더라도 구조와 사용 방식이 다르면 같은 위험 후보로 자동 확정하지 않는다.

## 9. `material_tags`

`material_tags`는 원재료, 제품, 부산물, 세척제, 연료, 가스 및 분진 등 취급 물질과 위험요인을 연결한다.

- 물질명, 물질군, 물리적 상태, 취급 형태 및 노출 경로를 구분한다.
- 물질 데이터는 최신 SDS 등 확인 가능한 근거의 버전과 검토일을 연결한다.
- 제품명만으로 유해성이나 필요한 조치를 확정하지 않는다.
- 혼합, 가열, 절단, 분사 등 작업 조건에 따라 달라지는 위험은 `work_type_tags`와 함께 후보화한다.

## 10. `environment_tags`

`environment_tags`는 장소와 작업환경 조건을 위험요인 후보에 연결한다.

주요 태그 축은 옥내·옥외, 밀폐·협소 공간, 고소, 지하, 경사·단차, 통행 동선, 환기, 조도, 소음, 진동, 온열·한랭, 강수·강풍, 분진 및 주변 동시작업 조건을 포함할 수 있다.

환경 태그는 센서값, 기상정보 또는 사용자 입력만으로 조치를 확정하지 않는다. 측정 시점, 측정 위치, 장비 상태 및 실제 작업 노출을 관리자가 함께 확인해야 한다.

## 11. `work_type_tags`

`work_type_tags`는 실제 수행 작업과 작업 단계를 위험요인 후보에 연결한다.

- 정상 운전뿐 아니라 설치, 시운전, 자재 반입, 운반, 적재, 청소, 정비, 해체 및 비상 대응을 포함한다.
- 단독작업, 교대·야간작업, 도급·혼재작업, 비정형·일회성 작업을 구분할 수 있어야 한다.
- 작업명, 작업 단계, 작업 빈도, 예상 시간, 투입 인원 및 숙련도는 후보 우선순위의 참고 신호로 사용한다.
- 작업유형 태그의 일치는 위험 존재나 위험 수준의 확정을 의미하지 않는다.

## 12. `company_hazard_candidates` 구조

`company_hazard_candidates`는 특정 회사·사업장에 제안된 위험요인 후보와 검토 흐름을 나타내는 논리 구조다.

| 논리 필드 | 설명 |
| --- | --- |
| `candidate_id` | 후보 식별자 |
| `company_key`, `site_id` | 회사 및 현장 범위 |
| `hazard_id` | `hazard_master` 연결값 |
| `candidate_title` | 회사 맥락을 반영한 후보 표시명 |
| `discovery_sources` | 업종·설비·물질·환경·작업·참여 등 발견 신호 |
| `matched_tags` | 일치한 태그와 일치 사유 |
| `source_record_refs` | TBM, 제보, 아차사고 등 원본 운영기록 연결값 |
| `candidate_score` | 정렬용 설명 가능한 후보 점수 |
| `confidence_band` | 낮음·보통·높음 등 검토 우선순위 구간 |
| `duplicate_of` | 기존 후보 또는 회사 원장 항목과의 중복 관계 |
| `worker_feedback_summary` | 참여 의견의 비식별 요약 |
| `representative_confirmation_status` | 근로자대표 참여확인 연결 상태 |
| `admin_review_status` | 신규, 검토중, 보완요청, 승인, 보류, 제외 |
| `admin_review_note` | 판단 근거와 후속 확인사항 |
| `promoted_hazard_ref` | 승인 후 회사별 원장 연결값 |
| `monthly_report_candidate` | 월간보고서 반영 검토 여부 |
| `next_assessment_candidate` | 다음 정기 위험성평가 재검토 여부 |
| `created_at`, `reviewed_at` | 생성 및 검토 시각 |

후보 점수는 정렬과 검토 지원에만 사용한다. 점수가 높다는 이유만으로 위험의 존재, 위험 수준, 조치 또는 법적 의무를 확정하지 않는다.

## 13. `worker_feedback` 기반 후보 승격 기준

근로자 참여 데이터는 현장에 반복되거나 새로 나타난 위험 신호를 보완하는 핵심 근거다. 후보 승격은 다음 기준을 조합해 관리자 검토 대상으로 올리는 것을 의미하며, 회사 원장 반영을 자동 승인하는 것은 아니다.

| 신호 | 승격 검토 기준 |
| --- | --- |
| 반복성 | 서로 다른 시점 또는 참여자에게서 유사 신호가 반복됨 |
| 구체성 | 위치, 작업, 설비, 물질 또는 발생 조건이 식별 가능함 |
| 직접성 | 작업자가 직접 관찰하거나 경험한 내용임 |
| 중대 가능성 | 즉시 확인이 필요한 사고·건강 영향 가능성이 제기됨 |
| 증빙 연결 | 사진, TBM, 점검 또는 기존 조치 기록과 연결 가능함 |
| 기존 원장 차이 | 기존 회사별 위험요인에 없거나 조건이 실질적으로 달라짐 |

단일 익명 제보도 중대 가능성이 있으면 횟수와 무관하게 우선 검토할 수 있다. 반대로 반복 횟수가 많더라도 자동 사실 판정은 하지 않는다. 익명성, 개인정보 최소화, 제보자 불이익 방지 및 접근 권한 분리를 운영 기준에 포함한다.

## 14. 근로자대표 참여확인과의 연결 기준

근로자대표 참여확인은 후보 목록을 일방적으로 승인받는 절차가 아니라, 현장 의견이 검토 과정에 제시되고 확인되었음을 기록하는 연결점이다.

1. 관리자 검토용 후보 중 현장 의견 확인이 필요한 항목을 참여확인 대상으로 묶는다.
2. 대표에게 후보 요약, 발견 근거, 관련 작업 및 의견 제출 경로를 제공한다.
3. 확인, 추가 의견, 이견, 정보 부족 상태를 구분해 기록한다.
4. 확인 시각과 대상 버전을 남겨 이후 변경된 후보와 혼동하지 않는다.
5. 대표의 확인을 위험성평가 완료, 조치 승인 또는 사업주 책임의 이전으로 해석하지 않는다.
6. 이견이나 보완 의견은 관리자 재검토 항목으로 되돌리고 원문과 처리 결과를 연결한다.

## 15. 관리자 검토 후 회사별 위험요인 원장화 기준

관리자는 후보의 실제 적용 여부를 현장 조건과 근거 기록으로 검토한다. 원장 승격 전 최소 확인 항목은 다음과 같다.

- 대상 회사, 사업장, 공정, 위치 및 작업 범위
- 관련 설비·물질·환경·작업 조건의 실제 존재 여부
- 기존 회사 위험요인과의 중복 또는 변경 관계
- 근로자 및 근로자대표 의견과 미해결 이견
- 확인한 근거, 검토자, 검토일 및 다음 검토 시점

승인된 후보는 회사별 위험요인 원장에 신규 항목 또는 기존 항목의 변경 제안으로 연결한다. 원장에는 마스터 버전, 회사별 설명, 적용 범위, 출처 후보, 상태 및 변경 이력을 보존한다. 제외·보류 후보도 사유와 함께 남겨 같은 후보가 반복 생성될 때 검토 맥락을 제공한다.

## 16. 월간보고서 연결 기준

월간보고서는 해당 기간의 위험요인 발견 및 검토 활동을 운영기록으로 요약한다.

포함 후보는 신규 발견 수, 출처별 후보 수, 근로자 의견 수, 대표 확인 상태, 관리자 검토 상태, 원장 승격 수, 보류·제외 수 및 다음 검토 항목이다. 보고서에는 원본 후보와 검토 기록을 추적할 수 있는 참조를 두되 고객 민감정보와 불필요한 개인정보는 제외한다.

후보 수나 검토 상태를 위험성평가 수행 실적, 안전 확보 또는 조치 이행의 확정치처럼 표시하지 않는다. 미검토·보완 필요 상태는 완료 항목과 명확히 구분한다.

## 17. 고객용 Export 연결 기준

고객용 Export는 회사·사업장·기간 권한 범위 안에서 정제된 위험요인 후보 및 검토 이력을 제공한다.

- 제공 후보: 후보명, 6대 참고 분류, 발견 경로 요약, 적용 작업·위치, 참여 의견 요약, 대표 확인 상태, 관리자 검토 상태, 원장 연결 상태, 검토일
- 기본 제외: 내부 점수 계산 상세, 내부 식별자, 원본 비공개 메모, 제보자 식별정보, 원시 입력 전체, 시스템 운영정보
- 상태 표현: “후보”, “검토중”, “보완 필요”, “관리자 승인” 등 실제 단계 그대로 표시
- 추적성: Export 기준일, 데이터 범위 및 포함된 후보 버전을 표시

Export는 고객이 운영 흐름을 확인하기 위한 자료이며 위험성평가서, 법률 의견서 또는 조치 확정서를 자동 대체하지 않는다.

## 18. 다음 정기 위험성평가 후보 재반영 기준

다음 정기 위험성평가 준비 시 아래 항목을 재검토 후보로 제공할 수 있다.

1. 관리자 승인 후 회사별 원장에 새로 연결된 위험요인
2. 보류되었으나 현장 조건 변화나 추가 참여 신호가 발생한 후보
3. 반복 제보, 아차사고 또는 TBM에서 재등장한 후보
4. 신규 설비·물질·공정·작업환경과 연결된 후보
5. 이전 평가 이후 마스터 근거나 분류가 변경된 후보
6. 조치 이후에도 잔여 위험 또는 새로운 작업 조건 확인이 필요한 항목

재반영 시 이전 검토 결과, 변경된 근거 및 현재 현장 조건을 함께 제시한다. 과거 승인이나 제외 상태를 그대로 복제하지 않으며, 다음 평가의 최종 대상과 판단은 관리자가 현장 상황에 따라 다시 결정한다.

## 19. AI 역할 제한

- AI는 후보 제안자다.
- AI는 법적 판단자나 조치 확정자가 아니다.
- 최종 판단과 책임은 관리자와 사업주에게 있다.

AI가 수행할 수 있는 역할은 태그 추천, 유사 후보 검색, 참여 의견 요약, 중복 후보 제안, 검토 우선순위 설명 및 누락 가능성 알림으로 제한한다. AI 결과에는 사용한 입력 범위와 제안 이유를 가능한 범위에서 표시하고, 관리자가 원문을 확인할 수 있어야 한다.

AI가 위험 존재, 위험 수준, 법률상 의무, 작업중지, 개선조치, 조치 완료, 대표 확인 또는 위험성평가 결과를 자동 승인해서는 안 된다. 낮은 품질의 입력, 근거 부족, 충돌하는 정보 및 모델 불확실성은 숨기지 않고 사람의 추가 확인 대상으로 표시한다.

## 20. 금지 표현

제품, 영업, 보고서, Export 및 AI 응답에서 다음 의미의 표현을 사용하지 않는다.

- SafeMetrica가 위험성평가를 수행·대행하거나 기존 대행 업무를 대체한다는 표현
- 특정 위험요인 개수를 공인된 전체 목록으로 단정하는 표현
- 서비스 이용만으로 법률상 절차나 의무가 모두 충족되었다는 표현
- 행정상 불이익이나 처벌을 막아 준다는 보장
- 사고 관련 책임을 없애거나 이전한다는 보장
- 재해가 발생하지 않는다는 보장
- AI가 위험, 조치, 승인, 완료 또는 적법성을 최종 결정한다는 표현
- 근로자대표 확인을 사업주의 책임 이전이나 위험성평가 결과 승인으로 해석하는 표현
- 내부 인증정보, 운영 비밀 또는 고객 민감정보를 설명·예시로 노출하는 표현

권장 표현은 “후보를 제안한다”, “확인이 필요하다”, “관리자 검토 대상으로 연결한다”, “운영기록으로 남긴다” 및 “다음 평가의 재검토 후보로 제공한다”이다.
