# SafeMetrica Supabase 안전운영 원장 현재 상태

- 기준일: 2026-06-08
- 문서 성격: 현재 상태 감사 기록
- 대상: SafeMetrica Supabase에 적재되는 업체별 안전운영 기록

## 1. 목적

이 문서는 Supabase에 적재되는 업체별 안전운영 기록의 2026-06-08 현재 상태를 기록한다.

SafeMetrica Supabase가 단순한 shadow-write 저장소를 넘어 업체별 안전운영 원장으로 확장 가능한지 판단하고, 향후 `operational_ledger`, 업체별 export view, 월간보고서 및 증빙 export를 설계하기 전에 기준 상태를 잠그는 것이 목적이다.

## 2. 현재 확인된 주요 테이블

현재 확인된 주요 테이블은 다음과 같다.

| 테이블 | 현재 확인된 역할 |
| --- | --- |
| `companies` | 업체 기준정보 |
| `sites` | 업체별 현장 기준정보 |
| `risk_instances` | 위험성평가 인스턴스 기록 |
| `actions` | 위험요인 등에 대한 조치 기록 |
| `evidence_items` | 조치 및 운영 증빙 기록 |
| `tbm_voice_submissions` | 업체별 TBM 음성기록 제출 원장 |
| `field_participation_submissions` | 업체별 공유확인·위험제보 등 현장참여 제출 원장 |
| `profiles` | 사용자 프로필 및 역할 연계 기준정보 |
| `cleanup_backup_field_participation_20260608` | 2026-06-08 현장참여 테스트 데이터 정리 전 백업 |

## 3. 현재 확인된 적재 상태

### 3.1 `field_participation_submissions`

| 업체코드 (`tenant_code`) | 업체명 | 제출유형 | 상태 | 현재 원본 건수 |
| --- | --- | --- | --- | ---: |
| `bubblemon` | 버블몬코리아 | 공유확인 | 접수 | 18건 |
| `hankookgreen` | 한국그린환경 | 공유확인 | 접수 | 1건 |

한국그린환경의 위험제보 테스트 데이터 19건은 `cleanup_backup_field_participation_20260608` 테이블에 백업한 뒤 `field_participation_submissions` 원본에서 삭제를 완료했다. 따라서 위 표의 현재 원본 건수에는 해당 위험제보 테스트 데이터가 포함되지 않는다.

### 3.2 `tbm_voice_submissions`

| 업체코드 (`company_code`) | 업체명 | 제출유형 | 제출 건수 | 파일 확인 |
| --- | --- | --- | ---: | --- |
| `hankookgreen` | 한국그린환경 | TBM 음성기록 | 6건 | `uploaded_files` 6건 중 6건 존재 |

## 4. 누락값 확인

업체 구분에 필요한 핵심 코드의 누락 여부는 다음과 같이 확인됐다.

| 테이블 | 확인 필드 | 누락 건수 |
| --- | --- | ---: |
| `field_participation_submissions` | `tenant_code` | 0건 |
| `tbm_voice_submissions` | `company_code` | 0건 |

현재 확인 범위에서는 두 제출 원장 모두 업체 단위 분리 및 export에 필요한 업체코드를 보유하고 있다.

## 5. Notion 연결값

다음 Notion 연결값의 적재를 확인했다.

- `field_participation_submissions`: `notion_page_id`, `notion_url`
- `tbm_voice_submissions`: `notion_page_id`, `notion_page_url`

Supabase row와 Notion page의 생명주기는 자동으로 동기화되지 않는다. Supabase row를 삭제해도 연결된 Notion page는 자동 삭제되지 않으므로, 테스트 데이터나 운영 데이터를 정리할 때 Notion page는 별도 대상 목록과 절차를 통해 정리해야 한다.

## 6. 중복 제출 이슈 및 조치

### 6.1 확인된 현상

한국그린환경 위험제보 **“차고지 동선분리”** 테스트에서 제출 의도는 1건이었으나 실제로는 18건이 생성됐다.

- `rows_count`: 18
- `distinct_notion_pages`: 18

각 row가 서로 다른 Notion page와 연결되어 있어, 단순 조회 중복이 아니라 제출 요청이 여러 번 처리된 것으로 확인됐다.

### 6.2 원인 판단

확인된 사용자 흐름과 생성 결과를 기준으로 다음 요인이 복합적으로 작용했을 가능성이 있다고 판단했다.

- 제출 버튼의 반응감 부족
- 제출 처리 중 버튼 또는 폼 잠금 부재
- 동일 입력에 대한 중복 submit 가능성

### 6.3 적용 조치와 결과

- `#357 fix: prevent duplicate field participation submissions`를 적용했다.
- 적용 후 신규 테스트 제출은 1건만 생성되어 클라이언트 측 1차 방어 효과를 확인했다.
- 기존 중복 18건과 조치 확인용 신규 1건을 합한 테스트 데이터 총 19건은 `cleanup_backup_field_participation_20260608` 테이블에 백업했다.
- 백업 완료 후 19건을 `field_participation_submissions` 원본에서 삭제했다.
- 추적과 복구 가능성을 위해 백업 테이블은 당분간 유지한다.

이 조치는 확인된 중복 제출 경로에 대한 1차 방어다. 향후 원장 설계에서는 `clientSubmissionId`의 유일성 또는 멱등성 보장 범위를 별도로 검토해야 한다.

## 7. 업체 구분 기준

현재 제출 원장별 업체 식별 필드가 서로 다르다.

| 원장 | 원본 업체 식별 필드 | export 정규화 필드 |
| --- | --- | --- |
| `field_participation_submissions` | `tenant_code` | `company_key` |
| `tbm_voice_submissions` | `company_code` | `company_key` |

원본 테이블에서는 기존 필드명을 유지하되, 통합 조회 및 export 기준에서는 두 값을 `company_key`로 정규화한다. 정규화 과정에서는 업체코드가 실제 `companies` 기준정보와 일치하는지도 검증할 필요가 있다.

## 8. 현재 원장 판단

2026-06-08 현재 Supabase는 업체별 운영기록 원장 후보로 정상 작동 중이다.

다음 근거를 확인했다.

- 업체코드를 포함한 현장참여 기록이 적재되고 있다.
- 업체코드를 포함한 TBM 음성기록과 업로드 파일이 적재되고 있다.
- Notion 연결 식별자와 URL이 함께 보존되고 있다.
- 중복 테스트 데이터는 별도 백업 후 원본에서 분리할 수 있었다.

다만 현재 데이터 구조는 하나의 통합 원장이 아니라 다음 영역별 원장으로 분리되어 있다.

- TBM 원장
- 현장참여 원장
- 위험성평가·조치·증빙 원장

따라서 업체별·기간별 안전운영 내역을 일관된 형식으로 조회하려면 `operational_ledger` 또는 이에 준하는 통합 export view 설계가 필요하다. 이 판단은 현재 적재 상태에 대한 것이며, 권한 정책·장기 보존·법적 요건 충족 여부를 보장하는 평가는 아니다.

## 9. export 가능 필드

현재 원장과 연결 데이터에서 업체별 export 후보로 검토할 수 있는 필드는 다음과 같다. 실제 컬럼명과 포함 범위는 Ledger schema v1 및 export view 설계에서 확정한다.

| export 후보 | 용도 또는 정규화 기준 |
| --- | --- |
| 업체코드 | `tenant_code` 또는 `company_code`를 `company_key`로 정규화 |
| 업체명 | 업체 표시명 |
| 제출유형 | 공유확인, 위험제보, TBM 음성기록 등 이벤트 유형 |
| 상태 | 접수, 처리 등 운영 상태 |
| 제목 | 제출 또는 기록 제목 |
| 내용 | 제출 본문, 설명 또는 기록 내용 |
| 위치 | 현장 또는 위험 위치 |
| 제출자 | 제출자 식별·표시 정보 |
| 익명 여부 | 익명 제출 여부 |
| 파일 URL | 첨부·업로드 파일 위치 |
| Notion page id | 연결된 Notion page 식별자 |
| Notion URL | 연결된 Notion page URL |
| `created_at` | 생성 시각 및 기간 필터 기준 후보 |
| `raw_payload` | 원본 제출 payload 보존값 |
| snapshot 계열 필드 | 제출 당시 업체·현장·사용자 등 기준정보 보존값 |
| `clientSubmissionId` | 클라이언트 제출 식별자 및 멱등성 검토 후보 |

파일 URL, 제출자 정보, `raw_payload`, snapshot 계열 필드는 개인정보·접근권한·보존정책을 확인한 뒤 export 범위를 정해야 한다.

## 10. 부족한 부분

통합 안전운영 원장과 업체별 export를 설계하기 전에 다음 사항을 보완하거나 결정해야 한다.

1. **통합 `event_id` 부재**: 원장 간 이벤트를 하나의 규칙으로 식별할 공통 키가 없다.
2. **업체코드 명칭 차이**: `tenant_code`와 `company_code`를 조회 계층에서 정규화해야 한다.
3. **관리자 조치상태 변경 이력 부족**: 현재 상태값뿐 아니라 누가, 언제, 무엇을 변경했는지 기록하는 상태 전이 원장이 필요하다.
4. **증빙 연결 기준 필요**: `evidence_items`와 현장참여의 `file_urls`를 동일 사건·조치 기준으로 연결할 규칙이 필요하다.
5. **월간보고서 후보 기준 필요**: 어떤 이벤트와 상태를 월간 실적·미처리·조치 완료 항목에 포함할지 기준이 필요하다.
6. **업체별 export view 필요**: 업체 및 기간 필터가 강제되는 읽기 전용 통합 view 설계가 필요하다.
7. **RLS 및 권한 정책 점검 필요**: 업체 간 데이터 격리, 관리자·Owner 권한 및 export 실행 권한을 점검해야 한다.
8. **Blob 파일 cleanup 정책 필요**: row 삭제, 보존기간 만료, 테스트 데이터 정리 시 연결 파일의 보존·삭제 기준이 필요하다.

## 11. 안전ON 자료에서 참고할 보완 항목

안전ON 자료에서 확인한 운영 관점을 SafeMetrica에 그대로 확정하지 않고, 다음 설계 검토 항목으로 참고한다.

- 근로자와 관리자의 역할 분리
- 업체별 링크 또는 “내 직장 찾기”를 대체할 업체 진입 구조
- 관리자 승인 및 근로자 명단 매칭 필요성
- TBM 전파 여부와 확인자 기록
- 위험제보를 누구나 작성할 수 있는 접근 구조
- 조치 후 사진과 조치 내용 기록
- 공지사항 및 교육 메뉴의 장기 도입 여부
- 업체별 월간 안전운영 파일 export
- 위험성평가표 출력지원 및 운영지원에 대한 표현 기준

제품 및 문서에서는 법적 면책, 초기대응 무료 제공 등 검증되지 않았거나 확정되지 않은 표현을 사용하지 않는다. 외부 제도 인정, 처벌 방지 또는 특정 기관의 승인·인정을 보장하는 표현도 사용하지 않는다.

## 12. 권장 다음 단계

다음 순서로 원장과 export 기능을 구체화한다.

1. **1차 — Supabase current ledger 상태 문서 확정**: 이 문서를 현재 상태 기준선으로 확정한다.
2. **2차 — Ledger schema v1 문서 작성**: 통합 이벤트, 업체키, 상태 이력, 증빙 연결 및 보존 기준을 정의한다.
3. **3차 — export view 설계**: 업체·기간 단위 정규화 필드와 조회 범위를 설계한다.
4. **4차 — read-only export API v1**: 서버 측 인증과 업체별 접근 통제를 포함한 읽기 전용 API를 설계·구현한다.
5. **5차 — 관리자/Owner Export 버튼**: 권한이 확인된 사용자에게만 업체별 export 진입점을 제공한다.
6. **6차 — 월간보고서와 export 연결**: 월간보고서 후보 기준과 원장 이벤트를 연결한다.
7. **7차 — RLS/권한 정책 점검**: 업체 격리, 역할별 조회 범위 및 파일 접근정책을 검증한다.

## 13. 범위 및 금지 사항

이 문서는 2026-06-08 현재 확인된 상태를 기록한 문서이며 다음 변경을 포함하지 않는다.

- SQL migration 생성 또는 실행
- 데이터베이스 스키마·row·view·function 변경
- 애플리케이션 코드, API 또는 화면 변경
- Notion page 또는 Notion 데이터 변경
- Supabase 설정, 인증, RLS 또는 Storage 변경
- 환경변수 변경

또한 다음 내용을 문서에 기록하거나 보장하지 않는다.

- 실제 API Key, service role 또는 환경변수 값
- 법적 면책 또는 처벌 방지
- KOSHA 등 외부 기관의 인정·승인 보장

이후 실제 구현이나 운영 변경은 별도 설계 검토, 권한 검토 및 승인 절차를 거쳐야 한다.
