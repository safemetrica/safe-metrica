# SafeMetrica New Customer Log and GitHub Boundary v1

기준일: 2026-06-14  
문서 성격: 운영 기준 / 고객정보 기록 경계 / GitHub 문서화 원칙  
적용 대상: 신규 고객 온보딩, Risk Share Pack, Full SafeMetrica 운영

---

## 1. 목적

이 문서는 신규 고객 또는 잠재 고객을 만날 때 어떤 정보는 GitHub에 남기고, 어떤 정보는 Notion·CRM·Supabase 등 별도 운영 영역에 남길지 구분하기 위한 기준 문서다.

GitHub는 제품의 공통 기준과 코드 기준을 관리하는 곳이다.  
고객별 세부 상담 내용, 담당자 정보, 계약 조건, 실제 현장 자료를 저장하는 곳이 아니다.

---

## 2. 기본 원칙

SafeMetrica 운영 정보는 아래처럼 분리한다.

1. GitHub  
   제품 공통 규칙, 코드, 표준 문서, 기능 기준, 금지 표현, 온보딩 체크리스트를 관리한다.

2. Notion 또는 내부 CRM  
   고객별 미팅 로그, 상담 내용, 세팅 메모, 진행상태, 담당자 연락정보를 관리한다.

3. Supabase  
   실제 운영 원장, 제출 기록, 공유확인, 위험제보, 근로자대표 확인, Export 기준 데이터를 관리한다.

4. Vercel  
   배포와 환경변수 설정을 관리한다. 환경변수 실제 값은 문서에 기록하지 않는다.

---

## 3. GitHub에 남길 수 있는 것

GitHub에는 고객이 바뀌어도 반복 적용 가능한 공통 기준만 남긴다.

- 제품 공통 규칙
- 기능별 Truth Map
- 온보딩 표준 절차
- 고객 handoff guide
- QR 포스터 표준 문구
- 개인정보 안내 기준
- Export 필드 매핑 기준
- 월간보고서 구성 기준
- 금지 표현 목록
- Partner Demo 기준
- 신규 고객 세팅 체크리스트
- Supabase-first schema 기준
- Owner Console 운영 기준
- IP 발명요약서 초안
- 영업 캡처 순서 가이드

---

## 4. GitHub에 남기지 않을 것

GitHub에는 아래 정보를 남기지 않는다.

- 실제 고객 민감정보
- 잠재 고객 미팅 상세 로그
- 담당자 이름, 휴대폰, 이메일
- 실제 계약금액 또는 개별 협상 조건
- 실제 위험성평가 원본
- 실제 현장 사진
- 실제 근로자 이름 또는 식별정보
- 실제 제보 내용 원문
- Owner Token
- API Key
- service role
- 환경변수 실제 값
- Supabase 접속정보
- Vercel 환경변수 화면
- GitHub secrets 화면
- 고객별 토큰 포함 URL
- 내부 디버그 메시지

---

## 5. 고객명 사용 기준

GitHub 공통 문서에서는 고객명을 최소화한다.

가능한 표현:

- 신규 고객
- 잠재 고객
- 도입 후보 사업장
- Risk Share Pack 도입 후보
- 50인 이상 사업장
- 물류업 고객
- 폐기물업 고객
- 제조업 고객

주의가 필요한 표현:

- 실제 회사명
- 담당자 실명
- 실제 현장명
- 실제 위험성평가표 제목
- 실제 사고 또는 제보 내용

예외적으로 고객명을 문서에 남겨야 하는 경우에는 공개 가능 여부와 필요성을 먼저 확인한다.

## 6. 신규 고객 운영 로그 저장 위치

신규 고객별 실제 진행 로그는 GitHub가 아니라 별도 운영 공간에 남긴다.

저장 후보:

1. Notion 고객 운영허브
2. 내부 CRM
3. 계약 관리 문서
4. Supabase customer onboarding ledger
5. 별도 내부 운영일지

기록 항목 후보:

- 미팅일
- 고객명
- 담당자
- 문의 경로
- 관심 상품
- 제공받은 자료
- 세팅 필요 항목
- 보류 사항
- 다음 액션
- 계약 여부
- 온보딩 상태

## 7. GitHub 문서 작성 시 일반화 기준

고객 사례를 GitHub 문서에 반영할 때는 아래처럼 일반화한다.

예시 1:

실제 표현:
특정 회사가 위험성평가표를 제공하면 공유팩으로 전환한다.

GitHub 표현:
고객이 기존 위험성평가표 또는 위험성평가 결과지를 제공하면 공유 대상 위험요인을 정리하고 고객 확인 후 QR 공유팩을 활성화한다.

예시 2:

실제 표현:
특정 현장 담당자에게 CSV를 전달한다.

GitHub 표현:
고객용 CSV Export는 내부 운영자가 내려받아 고객 지정 수신자에게 전달한다.

예시 3:

실제 표현:
특정 고객에게 3분 시연을 한다.

GitHub 표현:
신규 고객 미팅에서는 관리자 홈, 근로자 QR, 대표 확인, 접수함, 월간요약 순서로 시연한다.

---

## 8. 문서와 실제 기능 반영의 차이

GitHub 문서에 기준이 저장됐다고 해서 앱 기능이 자동으로 바뀌는 것은 아니다.

구분:

- 문서 반영: 운영자와 개발자가 따를 기준이 생김
- 코드 반영: 실제 앱 화면과 기능이 바뀜
- DB 반영: Supabase 또는 Notion 구조가 바뀜
- 배포 반영: Vercel Production에 실제 서비스가 반영됨
- 고객 운영 반영: 고객 세팅, QR, Export, 월간요약이 실제 운영에 적용됨

따라서 중요한 기준은 문서화 후 필요한 경우 별도 코드 PR, DB migration, 운영 세팅으로 연결한다.

---

## 9. Risk Share Pack 신규 고객 기준

Risk Share Pack 신규 고객은 아래 순서로 관리한다.

1. 상담 또는 문의 접수
2. 고객 자료 수령 여부 확인
3. 기존 위험성평가표 또는 결과지 보유 여부 확인
4. 공유 대상 범위 확인
5. 고객별 세팅 로그는 Notion 또는 내부 CRM에 기록
6. 공통 개선사항만 GitHub 문서 또는 코드로 반영
7. 실제 제출·확인 기록은 Supabase 원장에 저장

---

## 10. 금지 표현

GitHub 문서, 영업자료, 고객 안내문에서 아래 표현은 사용하지 않는다.

- 법적 의무 완료 보장
- 과태료 방지
- 중대재해 면책
- 무재해 보장
- 위험성평가 대행
- 안전관리대행
- AI가 법적 판단
- AI가 조치완료 확정
- QR 제출만으로 법적 의무 완료

---

## 11. 최종 기준

GitHub는 SafeMetrica의 제품 기준과 반복 가능한 운영 표준을 남기는 곳이다.  
고객별 실제 상담·계약·자료·현장정보는 GitHub가 아니라 별도 운영 영역에 남긴다.  
공통화할 수 있는 패턴만 GitHub 문서 또는 코드로 승격한다.
