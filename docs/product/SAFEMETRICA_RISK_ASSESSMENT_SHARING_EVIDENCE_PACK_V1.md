# SafeMetrica 위험성평가 공유·참여 증빙 패키지 v1

기준일: 2026-06-12

문서 위치: `docs/product/SAFEMETRICA_RISK_ASSESSMENT_SHARING_EVIDENCE_PACK_V1.md`

작업명: `docs: define risk assessment sharing evidence pack v1`

## 1. 문서 목적

이 문서는 현재의 `Risk Share Link`를 단순한 **위험성평가 공유확인 QR**에서 다음 운영기록을 함께 정리하는 **위험성평가 공유·근로자 참여증빙·근로자대표 참여확인 패키지**로 확장하기 위한 상품·기능·증빙 기준이다.

```text
현재 범위
- 위험성평가 내용을 QR 또는 링크로 공유
- 일반 근로자의 공유확인 또는 의견 없음 기록
- 위험제보·아차사고·개선제안 접수와 관리자 검토 기록

확장 범위
- 근로자 참여 및 의견제출 기록의 묶음 정리
- 근로자대표 참여확인 전용 기록
- 월간보고서 및 고객용 Export에서 참여확인 근거를 구분해 제공하는 증빙 패키지
```

이 문서는 코드 수정 전에 다음 기준을 잠그는 문서다.

1. 실제 운영 중인 기능과 아직 구현되지 않은 확장 범위를 구분한다.
2. 근로자, 근로자대표, 관리자 기록을 서로 다른 목적과 책임으로 분리한다.
3. 월간보고서와 고객용 Export에 포함할 후보 항목을 정의한다.
4. 개인정보 최소수집, 동의, 접근권한 및 보존정책의 후속 검토 기준을 정한다.
5. 법적 효과를 보장하거나 AI가 최종 판단하는 것처럼 설명하지 않도록 문구 기준을 정한다.

이 문서 자체는 법률 검토서, 법정 서식 또는 법적 효력 확인서가 아니다. 실제 적용 범위와 적정성은 사업장 상황에 따라 사업장과 전문가가 검토해야 하며, 최종 판단과 책임은 관리자와 사업주에게 있다.

## 2. 범위 구분 원칙

### 2.1 현재 구현 또는 기준화된 범위

아래 항목은 기존 현장참여 및 Risk Share Link 기준 문서에서 운영 가능하거나 운영 기준이 정리된 범위다. 고객사별 저장 구조와 화면 연결 상태가 다른 항목은 제한 운영으로 표시한다.

| 항목 | 현재 기준 | 증빙 패키지에서의 역할 |
| --- | --- | --- |
| 위험성평가 공유확인 QR | QR 또는 링크를 통해 위험성평가 내용을 확인하고 공유확인 기록을 남기는 흐름 | 일반 근로자 공유확인의 진입점 |
| 근로자 의견 없음 제출 | 추가 의견이 없는 경우에도 공유확인 기록으로 분리하는 기준 | 공유 사실과 의견 부재를 기록하되 조치완료로 해석하지 않음 |
| 위험제보·아차사고·개선제안 분기 | 제출 유형에 따라 관리자 검토대상을 구분하는 기준 | 근로자 참여 의견의 유형별 접수 근거 |
| 위치/구역만으로 위험제보 처리하지 않는 기준 | 위치 또는 구역 값만 있고 구체적인 위험 내용이 없으면 위험제보로 자동 확정하지 않음 | 오분류 방지 및 관리자 확인 근거 |
| 관리자 검토대상 분리 | 단순 공유확인과 검토가 필요한 의견·제보를 분리 | 관리자 검토 이력의 시작점 |
| 처리상태·조치메모 기록 | 접수, 검토 중, 조치 필요, 조치 완료, 보류·반려 등 운영 상태와 메모 기록 기준 | 검토 및 조치 경과의 운영기록 |
| 월간보고서 반영 후보 | 위험제보, 아차사고, 개선제안, 조치 경과 등을 보고서 후보로 연결 | 확정 보고가 아닌 관리자 검토용 초안 |
| 고객용 Export 후보 | 제출 원본, 분류, 검토 상태, 조치 상태, 메모 및 첨부 연결 후보 | 고객 전달용 운영기록 묶음 |
| Risk Share Link 문구 원장 기준 | 화면·영업·제안서 문구를 승인된 표현과 금지 표현으로 관리 | 패키지 전체 표현의 단일 기준 |

### 2.2 위치/구역 값의 해석

`location`, 위치 또는 구역 값은 작업 맥락을 찾기 위한 보조정보다. 다음 원칙을 적용한다.

- 위치/구역만 입력되었다는 이유로 위험제보, 아차사고 또는 개선제안으로 확정하지 않는다.
- 제출 유형, 제목, 내용, 확인 metadata 등 직접 근거를 우선한다.
- 직접 근거가 누락되거나 충돌하면 `관리자 검토대상`으로 분리한다.
- AI 또는 규칙 기반 분류는 후보를 제안할 수 있으나 원본 유형이나 조치 상태를 자동 확정하지 않는다.

### 2.3 아직 미완성인 범위

아래 항목은 이번 문서에서 기준만 제시하며 현재 제공 기능으로 설명하지 않는다.

| 미완성 항목 | 필요한 결과 | v1 상태 |
| --- | --- | --- |
| 근로자대표 참여확인 전용 구조 | 일반 근로자 공유확인과 분리된 입력·조회·권한·이력 구조 | 준비 중 |
| 근로자대표 성명/소속/직책/확인일/의견란 | 대표 확인 주체와 확인 맥락을 식별할 표준 필드 | 준비 중 |
| 이견 있음/없음 | 단순 확인과 이견 제출을 구분하고 후속 검토로 연결 | 준비 중 |
| 근로자대표 확인서/PDF | 확인 범위, 의견, 이견, 확인 주체를 요약한 출력물 | 준비 중 |
| 월간보고서 근로자대표 참여확인 별도 표시 | 일반 공유확인 수와 대표 확인 여부를 분리해 표시 | 준비 중 |
| 고객용 Export에서 근로자대표 확인 목록 분리 | 대표 확인 목록을 일반 근로자 기록과 별도 시트·섹션·파일로 구분 | 준비 중 |

근로자대표 확인 기록이 없거나 전용 구조가 연결되지 않은 상태에서는 `미확인`, `자료 없음`, `연결 전`처럼 사실에 맞게 표시한다. 일반 근로자의 공유확인 수, 관리자 검토 기록 또는 임의의 담당자 이름으로 근로자대표 참여확인을 대체하지 않는다.

## 3. 기능 상태표

### 3.1 상태 정의

| 상태 | 정의 | 외부 표현 원칙 |
| --- | --- | --- |
| 운영중 | 현재 정식앱 또는 기존 운영 흐름에서 실제로 사용할 수 있는 기능 | 제공 범위와 기록 의미를 한정해 설명 |
| 제한 운영 | 일부 고객사, 화면, 저장 구조 또는 데이터 조건에서만 연결 가능한 기능 | 제한 조건과 `후보`임을 함께 표시 |
| 준비 중 | 상품·기능 기준은 있으나 구현 또는 정식 제공 전인 기능 | 현재 기능처럼 표현하지 않고 `준비 중`으로만 표시 |
| 장기 검토 | 개인정보, 법률, 비용, 인증 또는 운영부담 검토가 선행되어야 하는 기능 | 도입을 약속하지 않고 검토 대상임을 표시 |
| 금지 표현 | 기능 유무와 관계없이 법적 효과, 책임 면제, 자동 판단 등을 단정하는 표현 | 화면, 영업자료, 제안서, 보고서에서 사용 금지 |

## 4. 참여 주체와 기록 분리

증빙 패키지는 최소한 다음 세 기록군을 구분한다.

| 기록군 | 주체 | 기록 목적 | 대체할 수 없는 항목 |
| --- | --- | --- | --- |
| 일반 근로자 공유확인 | 위험성평가 내용을 공유받은 근로자 | 공유·확인 및 의견제출 사실 정리 | 근로자대표 참여확인 |
| 위험제보·아차사고·개선제안 | 근로자 또는 선택 입력 제출자 | 위험·사건·개선 의견의 접수와 후속 검토 | 관리자 조치 적정성 판단 |
| 근로자대표 참여확인 | 해당 사업장의 기준에 따라 확인 주체로 정해진 근로자대표 | 확인 범위, 의견, 이견 및 확인일 정리 | 사업주의 최종 판단 또는 법률상 결론 |
| 관리자 검토·조치 기록 | 권한 있는 관리자 | 분류, 검토, 조치 필요 여부, 조치메모 및 상태 이력 관리 | 실제 현장 조치 자체 또는 법적 적정성 보장 |

하나의 사람이 여러 역할을 수행하더라도 제출 목적과 권한에 따라 기록을 구분한다. 일반 공유확인 제출에 직책 문자열이 있다는 이유만으로 근로자대표 확인으로 자동 전환하지 않는다.

## 5. 개인정보 입력 기준

### 5.1 공통 원칙

- 목적 달성에 필요한 최소정보만 수집한다.
- 입력 화면에는 개인정보 수집·이용 안내를 제공하고 사용자의 체크 기록이 필요하다.
- 안내에는 수집 항목, 이용 목적, 보유·이용 기준, 동의 거부 가능 여부 및 거부 시 제한되는 기능을 실제 정책에 맞게 포함하는 방안을 후속 검토한다.
- 개인정보와 익명 제보의 접근권한을 구분하고, 보고서와 Export에는 업무상 필요한 최소 범위만 노출한다.
- 보존기간, 정정·삭제, 열람권한, 다운로드 이력 및 파기 기준은 별도 개인정보 정책과 구현 검토를 따른다.
- 실제 고객 또는 근로자의 개인정보를 샘플 문서, Demo 데이터 또는 개발용 예시에 사용하지 않는다.

### 5.2 일반 근로자 공유확인

v1 입력 후보는 다음과 같다.

```text
필수 또는 사업장별 선택 후보
- 이름
- 소속 또는 작업조
- 휴대폰 뒤 4자리 또는 사번
- 개인정보 수집·이용 안내 체크
```

휴대폰 뒤 4자리와 사번 중 무엇을 사용할지는 사업장별 식별 필요성, 중복 가능성 및 개인정보 최소수집 원칙을 검토해 선택한다. 두 값을 항상 동시에 받는 것을 기본값으로 두지 않는다.

### 5.3 위험제보·아차사고·개선제안

- 익명 제출을 허용하거나 이름·소속 등 식별정보를 선택 입력으로 둘 수 있다.
- 익명 여부는 원본 제출값으로 보존하고 보고서나 Export에서 임의로 실명화하지 않는다.
- 후속 연락이 필요한 경우에도 연락처 전체 수집을 기본값으로 두지 않고 목적과 동의를 별도로 검토한다.
- 첨부파일에는 얼굴, 차량번호, 연락처 등 불필요한 개인정보가 포함될 수 있으므로 안내, 접근권한 및 보존 기준이 필요하다.

### 5.4 근로자대표 참여확인

v1 입력 후보는 다음과 같다.

```text
- 이름
- 소속
- 직책
- 확인일
- 확인 범위
- 의견
- 이견 여부
- 첨부파일
- 확인 처리자 또는 기록 주체
- 서명 또는 확인 체크 후보
- 개인정보 수집·이용 안내 체크
```

서명 이미지 또는 전자서명 기능은 단순 확인 체크와 개인정보·인증·법적 의미가 다를 수 있으므로, 구현 전 별도 검토 없이 동일한 효력을 부여하지 않는다.

### 5.5 v1 기본값에서 제외하는 항목

- 휴대폰번호 전체 수집은 v1 기본값이 아니며 장기 검토로 둔다.
- SMS 등 휴대폰 본인인증은 v1 기본값이 아니며 장기 검토로 둔다.
- 위치정보, 기기정보 또는 생체정보를 본인확인 수단으로 자동 수집하지 않는다.
- 장기 검토 항목을 도입하려면 수집 필요성, 대체수단, 동의, 보안, 비용, 접근성, 보존 및 파기 기준을 먼저 확정한다.

## 6. 근로자대표 참여확인 구조

### 6.1 필드 후보

아래는 후속 스키마·폼·Export 설계를 위한 논리 필드 후보이며, 이번 문서 작업에서 API나 DB 스키마를 변경하지 않는다.

| 필드 | 의미 | 입력·검증 후보 | 주의사항 |
| --- | --- | --- | --- |
| `representativeName` | 근로자대표 이름 | 문자열, 최소수집 및 권한 적용 | 일반 제출자명과 분리 |
| `representativeDepartment` | 소속 또는 부서 | 문자열 또는 사업장 기준 선택값 | 실제 조직 체계와 일치 여부 확인 |
| `representativeRole` | 직책 또는 대표 역할 | 문자열 또는 관리형 선택값 | 직책만으로 대표 자격을 자동 판단하지 않음 |
| `confirmedAt` | 참여확인 일시 또는 확인일 | 날짜·시간과 시간대 기준 | 제출 시각과 확인 대상 기간을 혼동하지 않음 |
| `confirmationScope` | 확인한 위험성평가, 현장, 기간 또는 문서 범위 | 식별자와 표시용 설명 후보 | 무엇을 확인했는지 재현 가능해야 함 |
| `opinion` | 근로자대표 의견 | 선택 또는 조건부 필수 장문 | 원문 보존, AI 요약과 구분 |
| `hasObjection` | 이견 있음 여부 | `true`/`false`/미선택 상태 검토 | 이견이 있으면 관리자 검토대상으로 연결 |
| `attachmentUrls` | 첨부 근거 연결 | 복수 파일 참조 후보 | 접근권한, 악성파일, 개인정보 검토 |
| `confirmedBy` | 확인 기록을 생성·확정한 주체 | 사용자 또는 관리자 식별 후보 | 대표 본인과 대리 입력자를 구분할 수 있어야 함 |
| `consentChecked` | 개인정보 수집·이용 안내 확인 | 체크 여부와 시점 후보 | 안내 문구 버전 또는 동의 이력 연계 검토 |

### 6.2 확인 흐름 후보

1. 확인 대상 위험성평가의 현장, 기간, 버전 또는 문서 범위를 표시한다.
2. 근로자대표의 이름, 소속, 직책을 입력하거나 권한 있는 프로필에서 불러온다.
3. 확인 범위와 확인일을 기록한다.
4. 의견과 이견 있음/없음을 입력한다.
5. 필요한 경우 첨부파일을 연결한다.
6. 개인정보 수집·이용 안내와 확인 체크를 기록한다.
7. 제출 후 일반 공유확인과 분리된 대표 확인 목록에 저장하는 구조를 후속 구현한다.
8. 이견 있음 또는 필수정보 누락 기록은 관리자 검토대상으로 표시한다.
9. 수정·철회·재확인이 발생하면 기존 기록을 덮어쓰기보다 변경 이력을 남기는 방안을 우선 검토한다.

### 6.3 이견 처리 원칙

- `hasObjection=true`는 위험성평가가 자동으로 무효이거나 미완료라는 법적 판단이 아니라, 관리자가 확인해야 할 이견이 있다는 운영 신호다.
- 이견 원문, 첨부, 접수 시점, 검토자, 검토 상태 및 조치메모를 연결할 수 있어야 한다.
- AI는 이견 내용을 요약하거나 분류 후보를 제안할 수 있지만 이견 해소, 조치 완료 또는 위험성평가 완료를 확정하지 않는다.
- 이견 있음 기록을 삭제하거나 단순 공유확인 수에 합산해 감추지 않는다.

### 6.4 확인서/PDF 후보

근로자대표 확인서 또는 PDF는 준비 중 기능으로 두며, 다음 항목을 포함하는 출력 구성을 검토한다.

- 회사·현장 및 대상 기간
- 확인 대상 위험성평가의 제목, 버전 또는 식별정보
- 근로자대표 이름, 소속, 직책
- 확인일과 확인 범위
- 의견 원문 또는 별첨 표시
- 이견 있음/없음
- 첨부 목록
- 확인 기록 생성 시점과 변경 여부
- `운영기록 요약이며 법적 판단은 별도 검토가 필요함`이라는 안내

PDF 생성 사실이 원본 입력의 진실성, 대표 자격, 조치 적정성 또는 법적 효력을 자동 보장하지 않는다.

## 7. 월간보고서 및 고객용 Export 반영 기준

### 7.1 공통 집계 원칙

- 회사, 현장, 대상 기간과 기준 시간대를 명시한다.
- 공유확인, 의견·제보, 관리자 검토·조치, 근로자대표 참여확인을 별도 지표로 구분한다.
- 같은 사건에서 파생된 여러 기록은 연결 식별자 후보를 사용해 중복 집계를 줄인다.
- 익명 제출과 개인정보는 보고 목적에 필요한 최소 범위로 표시한다.
- 데이터가 연결되지 않았거나 집계 기준이 확정되지 않았으면 임의의 숫자나 비율을 생성하지 않는다.
- AI 요약과 후보 분류는 원본 기록과 구분하며 관리자가 최종 보고 내용을 확인한다.

### 7.2 월간보고서 표시 후보

| 항목 | 표시 기준 | 해석 주의사항 |
| --- | --- | --- |
| 공유확인 수 | 대상 기간 내 유효한 일반 근로자 공유확인 기록 수 | 대상자 수 또는 확인율과 동일하지 않으며 중복 기준 필요 |
| 위험제보/아차사고/개선제안 검토대상 수 | 유형별 신규·검토 중 기록 수 | 위치/구역만으로 유형을 자동 확정하지 않음 |
| 조치완료 수 | 권한 있는 관리자가 조치완료로 기록한 수 | 실제 조치 적정성과 법적 완료를 보장하지 않음 |
| 근로자대표 참여확인 여부 | 대상 범위에 연결된 대표 확인 기록의 존재와 상태 | 일반 근로자 공유확인으로 대체하지 않음 |
| 이견 있음 여부 | 대표 확인 중 `hasObjection=true`인 기록 존재 여부 | 이견 해소 여부는 별도 상태와 메모로 관리 |
| 다음 달 확인 필요 항목 | 미확인, 이견 검토 중, 조치 미완료, 자료 보완 등 이월 후보 | 자동 확정이 아니라 관리자 확인 목록 |

근로자대표 참여확인은 월간보고서에서 최소한 다음처럼 별도 블록으로 표시하는 방안을 권장한다.

```text
근로자대표 참여확인
- 확인 상태: 확인 / 미확인 / 검토 필요 / 연결 전
- 확인일
- 확인 범위
- 이견: 있음 / 없음 / 미입력
- 의견 요약 후보
- 다음 확인 필요 항목
```

### 7.4 Export 주의사항

- Export는 운영기록 정리를 지원하며 기록의 진실성, 충분성, 대표 자격 또는 법률상 결론을 자동 확정하지 않는다.
- 다운로드 권한, 마스킹, 보존기간, 정정 이력, 생성 시각 및 생성 주체를 후속 설계에서 검토한다.
- 익명 제출자의 식별정보를 별도 데이터와 결합해 추정하거나 노출하지 않는다.
- 서명 이미지, 첨부파일 및 연락처가 포함될 경우 별도 접근권한과 안전한 전달 기준이 필요하다.

## 8. 문구 원장 기준

Risk Share Link와 증빙 패키지의 화면, 제안서, 계약 부속자료, Demo, 월간보고서 및 Export 설명은 기존 Risk Share Link 문구 원장 기준을 따른다. 새 문구가 필요하면 코드에 직접 산발적으로 추가하기 전에 문구 원장 문서에서 승인 표현과 금지 표현을 먼저 정리한다.

### 8.1 금지 표현

다음 표현과 같은 의미의 변형 문구를 사용하지 않는다.

- `법적 완료`
- `산안법 의무 충족 보장`
- `과태료 방지`
- `법적 면책`
- `중대재해 면책`
- `무재해 보장`
- `교육시간 인정 보장`
- `QR 제출만으로 법적 의무 완료`
- `AI가 조치완료 판단`
- `AI가 위험성평가 완료 판단`

금지 이유는 QR 제출, 확인 체크, PDF 생성, 관리자 상태 변경 또는 AI 분석만으로 사업장의 법적 의무 이행, 현장 조치의 적정성, 책임 결과를 단정할 수 없기 때문이다.

### 8.2 권장 표현

- `위험성평가 공유·확인 기록 지원`
- `근로자 참여 및 의견제출 기록 지원`
- `관리자 검토 이력 정리`
- `대표 확인용 운영기록 요약`
- `법적 판단은 사업장과 전문가 검토 필요`
- `AI는 후보 제안자이며 최종 판단은 관리자와 사업주`

상태가 제한 운영 또는 준비 중이면 권장 표현 뒤에도 실제 상태와 제한 조건을 함께 표시한다.

## 9. AI 사용 원칙

AI는 다음 작업의 후보 제안에 한해 사용할 수 있다.

- 의견·제보 유형 분류 후보
- 반복 위험 후보 탐색
- 관리자 검토 우선순위 후보
- 월간보고서 요약 초안
- 대표 의견 및 이견 요약 초안
- 누락 가능 필드 또는 다음 확인 필요 항목 후보

AI는 다음 항목을 최종 확정하지 않는다.

- 법률상 의무 이행 여부
- 근로자대표 자격 또는 참여의 법적 적정성
- 위험성평가 완료 여부
- 조치 필요 여부와 조치 완료의 최종 판단
- 현장 위험의 허용 가능성
- 책임 소재, 제재 회피 또는 면책 여부

AI 결과는 원본 기록과 구분해 표시하고, 관리자가 근거를 확인·수정·채택 또는 제외할 수 있어야 한다. 최종 판단과 책임은 관리자와 사업주에게 있다.

## 10. v1 완료 기준과 비범위

### 10.1 이 기준 문서의 완료 조건

- 현재 구현 또는 기준화된 범위와 미완성 범위가 분리되어 있다.
- 기능이 운영중, 제한 운영, 준비 중, 장기 검토, 금지 표현으로 구분되어 있다.
- 일반 근로자와 근로자대표의 입력 기준이 분리되어 있다.
- 근로자대표 참여확인 필드 후보와 이견 처리 원칙이 정의되어 있다.
- 월간보고서와 고객용 Export 후보 항목이 정의되어 있다.
- 금지 표현, 권장 표현 및 AI 역할 한계가 명시되어 있다.

### 10.2 이번 PR의 비범위

- TS/TSX 파일 수정
- 화면, 폼 또는 라우트 구현
- API 수정 또는 신규 생성
- DB 스키마, migration 또는 저장 로직 변경
- Notion 또는 Supabase 연동 변경
- 월간보고서 및 Export 실제 컬럼 추가
- 확인서/PDF 생성 구현
- 인증, 전자서명 또는 휴대폰 본인인증 구현
- 토큰, 인증정보, 비밀값 또는 환경변수 값 문서화
- 확인되지 않은 법령 조문, 수치 또는 기한의 제시

## 11. 후속 PR 방향

다음 작업은 각각 별도 PR로 분리하고, 구현 전 현재 저장 구조와 개인정보·권한 기준을 다시 확인한다.

   - 대표 확인 여부, 확인일, 이견 여부 및 다음 확인 필요 항목을 별도 표시한다.

   - 실제 구현 상태에 맞춰 계약·제안 문구를 문구 원장과 정합화한다.
