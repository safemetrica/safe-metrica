## 1. 문서 목적

이 문서는 #402에서 정리한 `SAFEMETRICA_RISK_ASSESSMENT_SHARING_EVIDENCE_PACK_V1.md`의 후속 문서다. 상위 문서가 Risk Share Link를 **위험성평가 공유·근로자 참여증빙·근로자대표 참여확인 패키지**로 확장하기 위한 전체 기준을 정의했다면, 이 문서는 그중 근로자대표 참여확인 기록을 실제로 구현하기 전에 필요한 세부 기준을 잠근다.

이번 문서에서 잠그는 범위는 다음과 같다.

1. 근로자대표 참여확인 기록의 논리 필드와 필수 여부 후보
2. 관리자 검토 상태값과 상태 변경 원칙
3. 이견 원문, 관리자 검토 및 AI 요약 후보의 분리 기준
4. 입력·조회·검토·Export 권한 기준
5. 생성·수정·검토·Export·월간보고서 반영 이력 기준
6. 월간보고서와 고객용 Export의 최소 반영 기준
7. 과장 또는 법률상 결론으로 오해될 수 있는 표현의 금지 기준

이 문서는 제품·운영 설계 기준이며 법률 검토서, 법정 서식, 근로자대표 자격 확인서 또는 의무 이행 확인서가 아니다. 실제 적용 전 사업장 상황과 관련 기준에 대한 별도 검토가 필요하며, 최종 판단과 책임은 관리자와 사업주에게 있다.

## 2. 근로자대표 확인 기록의 역할

### 2.1 일반 근로자 공유확인과 분리

일반 근로자 공유확인은 위험성평가 내용을 공유받고 확인했다는 운영기록이다. 근로자대표 참여확인은 특정 위험성평가와 확인 범위를 기준으로 대표의 확인, 의견 또는 이견을 별도 기록하는 흐름이다.

따라서 두 기록은 다음 기준으로 분리한다.

- 입력 화면, 목록, 집계 및 상세 조회 목적을 분리한다.
- 일반 근로자 공유확인 수로 근로자대표 확인을 대체하지 않는다.
- 근로자대표 확인 기록을 일반 공유확인 통계에 중복 합산하지 않는다.
- 대표 확인 기록이 없으면 `미확인`, `자료 없음` 또는 `연결 전`처럼 실제 상태를 표시한다.

### 2.2 위험제보·아차사고·개선제안과 분리

근로자대표의 `opinion` 또는 `objectionDetail`은 위험제보, 아차사고 또는 개선제안 접수 기록과 목적이 다르다. 대표가 의견이나 이견을 남겼다는 이유만으로 기존 제보 유형을 자동 생성하거나 확정하지 않는다.

- 대표 의견과 이견은 대표 확인 원장에 원문으로 보존한다.
- 별도 위험제보나 개선조치 접수가 필요하면 관리자가 원본과 연결되는 별도 기록을 생성한다.
- AI 또는 규칙 기반 분류는 연결 필요성의 후보만 제안할 수 있다.
- 익명 제보 기록과 대표 확인 기록을 결합해 익명 제출자를 추정하지 않는다.

### 2.3 관리자 조치완료와 분리

근로자대표 확인과 관리자 조치완료는 서로 다른 상태축이다.

- `reviewStatus`는 대표 확인 기록에 대한 관리자 검토 상태다.
- 위험요인 또는 제보에 대한 조치 상태는 별도 조치 원장의 상태를 따른다.
- 대표 확인, 이견 없음 또는 관리자 검토 완료를 조치완료로 변환하지 않는다.
- 조치 상태를 대표 확인 화면에 함께 보여줄 경우에도 출처와 상태명을 명확히 분리한다.

### 2.4 법적 효과에 대한 한계

근로자대표 확인 기록은 참여·확인·검토 이력을 정리하기 위한 운영기록이다. 이 기록만으로 법률상 의무 이행, 근로자대표 자격의 적정성, 위험성평가의 충분성, 현장 조치의 적정성 또는 책임 결과를 확정하지 않는다.

## 3. 필드 정의

### 3.1 공통 해석 원칙

- 아래 타입은 구현 전 논리 타입 후보이며 실제 DB 타입을 확정하지 않는다.
- `필수`는 저장을 허용하지 않는 강제값 후보, `조건부 필수`는 특정 상태나 입력에 따라 필요한 값 후보를 뜻한다.
- 화면 표시 여부는 기본 권한 화면을 기준으로 하며 역할과 맥락에 따라 마스킹할 수 있다.
- Export 및 월간보고서 포함 여부는 기본 후보값이다. 실제 출력 시 고객 권한, 대상 기간, 최소수집 원칙 및 관리자 확인을 다시 적용한다.
- 이름, 소속, 직책, 의견, 이견 상세, 첨부 및 처리 주체는 개인정보 또는 조직 내부정보가 될 수 있으므로 목적 외 조회와 재사용을 제한한다.
- 원본 입력값과 관리자 입력값, AI 요약 후보를 같은 필드에 덮어쓰지 않는다.

### 3.3 구현 전 추가 결정이 필요한 보조 metadata

위 논리 필드 외에도 실제 구현 PR에서는 다음 보조 metadata를 별도 설계해야 한다.

- 레코드 고유 ID, 생성 시각, 생성 주체, 마지막 수정 시각
- 동의 안내 문구 버전과 동의 일시
- 위험성평가 제목·버전의 제출 당시 snapshot 또는 참조 무결성 기준
- 첨부 파일명, MIME 유형, 크기, 저장 참조, 검사 상태
- 상태 변경 전후 값, 변경 주체, 변경 일시, 변경 사유
- Export/월간보고서 생성 ID, 생성 시각, 생성 주체, 포함 여부 snapshot
- 정정·철회 요청과 원본 보존 방식

보조 metadata는 이 문서의 20개 업무 필드를 대체하지 않으며, 감사 가능성과 시점별 재현성을 위해 추가되는 시스템 정보다.

## 4. 상태값 정의

### 4.1 `reviewStatus` 후보

`reviewStatus`는 **관리자가 근로자대표 확인 기록을 검토하는 운영 상태**다. 법적 판단 상태, 위험성평가 완료 상태 또는 현장 조치완료 상태가 아니다.

| 상태값 | 의미 | 진입 조건 후보 | 후속 처리 후보 |
| --- | --- | --- | --- |
| `미확인` | 관리자 확인이 아직 시작되지 않았거나 대표 확인 기록이 연결되지 않은 상태 | 레코드 초기 생성 또는 연결 전 | 관리자가 원본·연결 범위·권한을 확인 |
| `확인` | 대표 제출 기록이 접수되었고 기본 필드가 확인된 상태 | 필수 필드와 연결 대상의 기본 확인 | 이견 여부에 따라 검토 완료 또는 추가 검토로 이동 |
| `검토 필요` | 누락, 불명확한 범위, 첨부 확인 또는 추가 판단이 필요한 상태 | 자동 규칙 또는 관리자 판단 | 관리자 상세 검토, 보완 요청 또는 다른 상태로 변경 |
| `이견 검토 중` | `hasObjection=true`인 원문과 관련 자료를 관리자가 검토 중인 상태 | 이견 제출 확인 후 관리자 검토 시작 | 검토 메모 기록, 보완 요청 또는 검토 완료 |
| `보완 요청` | 대표 또는 내부 담당자에게 확인 범위·자료·설명의 보완을 요청한 상태 | 보완 항목과 사유가 특정됨 | 보완 수신 후 검토 필요 또는 이견 검토 중으로 재진입 |
| `검토 완료` | 대표 확인 기록에 대한 현재 관리자 검토가 끝난 상태 | 원본, 이견 및 필요한 보완을 검토하고 메모를 기록 | 후속 과제가 있으면 별도 조치 원장에 연결 |
| `반려` | 잘못된 대상 연결, 중복, 권한 없는 제출 등으로 현재 기록을 유효한 대표 확인 기록으로 사용하지 않는 상태 | 반려 사유와 근거 기록 | 원본 보존, 필요 시 올바른 기록 재제출 안내 |

### 4.2 상태 적용 원칙

- 초기 상태는 `미확인`을 후보로 하되, 제출 성공과 관리자 접수를 분리할 필요가 있으면 구현 PR에서 생성 상태를 추가 검토한다.
- `hasObjection=true`이면 최소한 관리자 검토대상으로 표시하고, 검토를 시작하면 `이견 검토 중`을 사용한다.
- `hasObjection=false`라는 이유만으로 자동으로 `검토 완료`로 변경하지 않는다.
- `검토 완료`는 대표 확인 기록에 대한 검토 종료일 뿐, 연결된 위험요인에 대한 조치완료를 뜻하지 않는다.
- `반려`는 원본 삭제를 뜻하지 않는다. 반려 사유와 상태 이력을 보존한다.
- 상태를 되돌리거나 재검토할 수 있어야 하며, 최신 상태만 남기고 과거 상태를 덮어쓰지 않는다.
- AI는 상태 후보를 제안할 수 있으나 상태를 최종 확정하거나 변경하지 않는다.

```text

확인

  → 확인
  → 이견 검토 중
  → 보완 요청
  → 검토 완료
  → 반려

이견 검토 중
  → 보완 요청
  → 검토 완료
```

## 5. 이견 처리 흐름

### 5.1 기본 흐름

1. 대표가 `hasObjection=true`를 선택하면 `objectionDetail` 입력을 조건부 필수로 안내한다.
2. 제출된 이견 기록을 관리자 목록에서 검토대상으로 명확히 표시한다.
3. `objectionDetail` 원문과 첨부를 제출 시점 기준으로 보존한다.
4. AI를 사용하는 경우 원문과 구분된 **요약 후보**만 생성한다.
5. 관리자가 원문, 확인 범위, 첨부 및 관련 위험성평가를 검토한다.
6. 관리자가 `reviewStatus`, `reviewMemo`, `reviewedBy`, `reviewedAt`을 기록한다.
7. 별도 위험요인 조치가 필요하면 대표 확인 원장을 덮어쓰지 않고 조치 원장에 참조 연결한다.
8. 월간보고서나 Export에는 권한과 최소 공개 원칙에 따라 원문 대신 이견 여부, 검토 상태, 메모 또는 요약 후보를 선택한다.

### 5.2 원문과 AI 요약 후보 분리

- AI는 `opinion` 또는 `objectionDetail`의 요약 후보를 제안할 수 있다.
- AI 요약 후보에는 생성 시각, 사용 모델 또는 처리 버전, 원본 참조, 관리자 채택·수정·제외 여부를 추적할 수 있어야 한다.
- AI 요약 후보를 원문 필드에 저장하거나 원문을 대체하지 않는다.
- 관리자가 채택한 요약도 대표가 직접 작성한 문장으로 표시하지 않는다.
- AI는 대표 의견의 의미, 정당성, 해결 여부 또는 법률상 효과를 최종 판단하지 않는다.

### 5.3 이견 유무의 해석 한계

- 이견 있음이 자동으로 위험성평가 미완료를 의미하지 않는다.
- 이견 없음이 위험성평가의 법률상 완료나 충분성을 의미하지 않는다.
- 이견에 대한 관리자 검토 완료가 관련 현장 조치완료를 의미하지 않는다.
- 이견이 별도 제보·개선과제로 연결되더라도 두 원장의 상태와 이력을 독립적으로 관리한다.

### 6.2 목록과 상세 분리

- 일반 근로자 공유확인 목록과 근로자대표 확인 목록을 데이터·화면·Export에서 논리적으로 분리한다.
- 대표 확인 기록의 상세 내용은 관리자 및 해당 대표에게 필요한 범위에서만 제공한다.
- 일반 근로자 화면에는 대표 이름, 의견 원문, 이견 상세 또는 관리자 내부 메모를 노출하지 않는다.
- 대표용 화면에 관리자 메모를 제공할 때는 외부 공개 가능한 내용과 내부 메모를 분리한다.
- 고객용 Export에서는 업무 목적에 필요한 최소 항목만 노출하고, 원문·계정 ID·내부 코드·직접 첨부 URL은 기본 제외 후보로 둔다.

### 6.3 개인정보 및 비밀정보 기준

- 이름, 소속, 직책, 의견, 이견, 첨부 및 처리 주체의 수집 목적과 보존 기준을 구현 전 검토한다.
- 익명 제보와 대표 확인 기록을 결합해 제출자의 신원을 추정하지 않는다.
- 다운로드 파일에는 접근권한, 생성 주체, 생성 시각, 대상 기간 및 재전달 주의사항을 둘 수 있다.
- 첨부는 인증된 다운로드 또는 만료되는 참조 방식을 검토하고 공개 URL을 문서나 Export에 그대로 싣지 않는다.
- 토큰, API Key, 비밀값 또는 환경변수 값은 문서, 화면, 로그, Export에 기록하지 않는다.

### 7.2 원본 보존과 정정 원칙

- 대표가 제출한 `opinion`과 `objectionDetail` 원문은 관리자가 직접 덮어쓰지 않는다.
- 정정이 필요하면 변경 전 원본과 정정본을 구분하고 정정 주체·사유·일시를 남긴다.
- 상태나 메모가 변경되어도 과거 값을 감사 이력에서 재구성할 수 있어야 한다.
- 삭제 요청, 보존기간 만료 및 법적 보존 필요성이 충돌하는 경우는 별도 정책과 권한 승인 절차를 따른다.
- Export와 월간보고서는 생성 시점의 포함 대상을 snapshot으로 남겨 이후 원장 변경과 구분할 수 있어야 한다.
- 감사 로그 자체에도 tenant 경계와 최소 접근권한을 적용한다.

## 8. 월간보고서 반영 기준

### 8.1 별도 섹션 원칙

월간보고서에는 일반 근로자 공유확인 수, 의견·제보 현황, 관리자 조치 현황과 분리된 **근로자대표 참여확인** 섹션을 둔다. 기록이 없으면 일반 공유확인이나 관리자 활동으로 대체하지 않고 `미확인`, `자료 없음` 또는 `해당 기간 연결 기록 없음`으로 표시한다.

### 8.2 표시 항목 후보

| 항목 | 표시 기준 후보 |
| --- | --- |
| 근로자대표 확인 상태 | 대표 확인 기록 존재 여부와 현재 운영 상태를 사실대로 표시 |
| 확인일 | `confirmedAt`의 날짜를 보고서 시간대 기준으로 표시 |
| 확인 범위 | `confirmationScope`를 원문 또는 관리자 확인 요약으로 표시 |
| 이견 여부 | `hasObjection`을 `있음`/`없음`/`확인 필요`로 표시 |
| 검토 상태 | 최신 `reviewStatus`를 표시하되 관리자 검토 상태임을 명시 |
| 다음 확인 필요 항목 | 보완, 재확인, 이견 검토, 별도 조치 연결 등 관리자 확인 항목 |
| 의견 요약 후보 | `opinion`·`objectionDetail`의 관리자 작성 또는 채택 요약. AI 생성 시 후보임을 표시 |

### 8.3 포함 및 제외 기준

- 기본적으로 `monthlyReportIncluded=true`이고 대상 기간·회사·현장·위험성평가가 보고서 범위와 일치하는 기록을 후보로 한다.
- 포함 플래그만으로 자동 확정하지 않고 생성 전 관리자 검토 단계를 둔다.
- 이름, 상세 소속, 원문 의견, 원문 이견, 첨부 URL 및 내부 계정 ID는 기본 요약본에서 제외한다.
- 이견 건수나 상태를 표시할 때 모수가 작아 개인을 쉽게 추정할 수 있으면 집계 수준을 조정한다.
- AI 요약은 원문과 대조 가능한 후보로 제공하고, 관리자 채택 전 확정 문구로 출력하지 않는다.
- 보고서 생성 후 포함 대상과 표시값 snapshot을 이력으로 남긴다.

## 9. 고객용 Export 반영 기준

### 9.1 Export 후보 컬럼

| Export 컬럼 | 연결 필드·출력 기준 후보 |
| --- | --- |
| 회사/현장 | `relatedCompanyCode`에 대응하는 안전한 회사 표시명 + `relatedSiteName` |
| 대상 기간 | Export 요청 기간 또는 연결된 위험성평가 적용 기간 |
| 위험성평가 식별정보 | `relatedRiskAssessmentId`에 연결된 제목·버전·안전한 표시 ID |
| 대표 이름 | `representativeName`; 권한과 전달 목적에 따라 마스킹 검토 |
| 대표 소속 | `representativeDepartment` |
| 대표 직책 | `representativeRole` |
| 확인일 | `confirmedAt` |
| 확인 범위 | `confirmationScope` |
| 이견 여부 | `hasObjection` |
| 검토 상태 | `reviewStatus`; 관리자 검토 상태임을 헤더 또는 안내문에 명시 |
| 검토 메모 | `reviewMemo` 중 고객 전달이 승인된 최소 내용 |
| 첨부 목록 | 파일명·개수·별도 Manifest 등 안전한 참조. 직접 공개 URL 제외 |
| 개인정보 안내 체크 여부 | `consentChecked`; 동의 내용 전체나 민감한 시스템 metadata는 제외 |

### 9.2 생성 기준

- 일반 근로자 공유확인 목록과 근로자대표 확인 목록을 별도 시트·섹션·파일로 구분한다.
- 기본적으로 `exportIncluded=true`이고 요청한 tenant·회사·현장·기간 범위와 일치하는 기록만 후보로 한다.
- Export 생성 권한, 대상 범위, 포함 레코드, 생성 일시와 생성 주체를 기록한다.
- 원문 의견과 이견 상세는 기본 Export 컬럼에서 제외하고, 업무상 필요한 경우 별도 권한과 전달 사유를 확인한다.
- 첨부 파일은 목록과 실제 파일 전달을 분리하고, 실제 파일은 별도 접근통제와 안전한 전달 방식을 사용한다.
- 반려 기록을 포함할 경우 반려 상태와 사유를 명확히 표시해 정상 확인 기록으로 오인되지 않게 한다.
- 정정 이력이 있는 값은 최신값 사용 여부와 생성 시점 snapshot을 추적할 수 있어야 한다.

### 9.3 Export 한계 안내

Export는 고객이 운영기록을 정리하고 내부 검토하는 것을 지원하는 자료다. Export 생성 또는 특정 컬럼의 존재만으로 법적 효력, 근로자대표 자격, 조치 적정성, 위험성평가의 충분성 또는 의무 이행 완료를 보장하지 않는다.

## 10. 금지 표현

화면, 안내문, 월간보고서, Export, 영업자료, 계약 부속자료 및 AI 결과에서 다음 표현과 같은 의미의 변형 문구를 사용하지 않는다.

- `법적 완료`
- `산안법 의무 충족 보장`
- `과태료 방지`
- `법적 면책`
- `중대재해 면책`
- `무재해 보장`
- `근로자대표 확인만으로 위험성평가 완료`
- `AI가 근로자대표 의견을 확정 판단`
- `AI가 이견 해소 판단`

위 표현은 대표 확인, 관리자 상태 변경, PDF·Export 생성 또는 AI 분석만으로 법률상 효과, 의무 이행, 위험성평가의 충분성, 조치 적정성 또는 책임 결과를 단정할 수 없기 때문에 금지한다.

### 10.1 권장 표현 후보

- `근로자대표 참여확인 운영기록`
- `대표 의견 및 이견 기록 지원`
- `관리자 검토 상태`
- `월간보고서용 요약 후보`
- `고객용 운영기록 Export`
- `AI는 요약 후보를 제안하며 최종 검토는 관리자와 사업주가 수행`
- `실제 적용과 법률상 판단은 사업장 및 전문가 검토 필요`

## 11. 구현 전 확인사항

후속 구현 PR은 최소한 다음 질문에 답한 뒤 진행한다.

1. 근로자대표 권한을 누가 어떤 근거와 유효기간으로 부여·회수하는가?
2. 대표 이름과 계정 주체가 다른 경우 화면과 이력에서 어떻게 표시하는가?
3. 위험성평가가 개정된 경우 기존 확인 범위와 새 버전의 관계를 어떻게 보존하는가?
4. 동의 안내 문구 버전과 확인 일시를 어디에 저장하는가?
5. 대표 원문 정정, 제출 철회, 중복 제출 및 잘못된 대상 연결을 어떻게 처리하는가?
6. 내부 검토 메모와 대표에게 공개할 메모를 별도 필드로 분리할 것인가?
7. AI 요약 후보를 어떤 별도 구조에 저장하고 채택·수정 이력을 어떻게 남기는가?
8. 첨부의 업로드 제한, 검사, 보존기간, 다운로드 권한 및 Export 전달 방식을 어떻게 정하는가?
9. 월간보고서와 Export의 포함 플래그를 레코드 전역 값으로 둘지 생성 건별 결정으로 둘지 어떻게 구분하는가?
10. tenant·회사·현장 경계와 예외 운영자 접근을 어떤 서버 권한 검사로 보장하는가?
