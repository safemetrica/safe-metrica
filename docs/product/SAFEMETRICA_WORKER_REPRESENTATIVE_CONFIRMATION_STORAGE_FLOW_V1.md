## 1. 문서 목적

이 문서는 근로자대표 참여확인 폼을 구현하기 전에 **저장 위치, 접근 경로, 권한 경계, 논리 원장 구조, 관리자 검토, 월간보고서 및 고객용 Export 연결 흐름**을 잠그기 위한 제품·운영 설계 기준이다.

상위 기준은 다음 문서를 따른다.

- `SAFEMETRICA_RISK_ASSESSMENT_SHARING_EVIDENCE_PACK_V1.md`: 일반 공유확인, 의견·제보, 관리자 조치 및 근로자대표 참여확인의 분리 원칙
- `SAFEMETRICA_WORKER_REPRESENTATIVE_CONFIRMATION_FIELDS_V1.md`: 근로자대표 참여확인 필드, 상태, 권한 및 이력 기준

이번 문서가 잠그는 범위는 다음과 같다.

1. 저장 구조 후보별 장단점과 v1 권장안
2. 대표 입력, 관리자 검토, 월간보고서 및 Export 접근 경로 후보
3. 제출부터 출력 후보 생성까지의 데이터 이동과 권한 경계
4. 별도 논리 원장, 원본 보존, 상태 변경 및 감사 이력 원칙
5. 중복, 정정, 이견 및 AI 보조 처리 원칙
6. 월간보고서와 고객용 Export의 분리 연결 기준

이 문서는 법률 검토서, 법정 서식 또는 근로자대표 자격 확인서가 아니다. 기록의 존재만으로 대표 자격, 위험성평가의 충분성, 현장 조치의 적정성 또는 법률상 결과를 판단하지 않는다.

## 2. 저장 원칙

### 2.1 일반 공유확인과 분리

근로자대표 참여확인은 일반 근로자의 위험성평가 공유확인과 목적, 입력 주체, 검토 상태 및 출력 권한이 다르므로 같은 제출 유형처럼 섞지 않는다.

- 근로자대표 확인은 논리적으로 별도 원장으로 본다.
- 일반 공유확인 행에 대표 여부 필드만 추가해 대표 확인으로 재해석하지 않는다.
- 일반 근로자 `field participation` 제출 수, 공유확인 수 및 조치 KPI에 합산하지 않는다.
- 대표 이름·소속·직책, 의견 원문, 이견 원문 및 관리자 메모를 일반 근로자 목록에 노출하지 않는다.
- 대표 확인이 없거나 연결되지 않은 경우 실제 상태를 `미확인`, `자료 없음` 또는 `연결 전` 등으로 표시한다.

### 2.2 물리 저장소와 논리 원장의 분리

`별도 원장`은 v1에서 반드시 별도 물리 데이터베이스 제품이나 테이블만을 의미하지 않는다. 초기 구현은 현재 앱 구조에 맞는 최소 저장 경로를 선택할 수 있으나, 다음 경계는 저장 매체와 무관하게 유지한다.

- 고유한 대표 확인 기록 식별자
- 위험성평가 및 회사·현장 연결 식별자
- 대표 제출 원문과 관리자 입력값의 분리
- `reviewStatus`와 상태 변경 이력
- 정정·철회·대체 관계
- 월간보고서 및 Export 포함 snapshot
- 접근 주체와 시각을 남길 수 있는 감사 이력

물리 저장 방식을 바꾸더라도 위 논리 계약을 유지해 Supabase-first 구조로 전환할 수 있어야 한다.

### 2.3 원본 우선과 파생값 분리

- 대표가 제출한 의견·이견 원문은 파생 요약과 별도로 보존한다.
- 관리자 메모와 검토 상태는 대표 원문 필드에 덮어쓰지 않는다.
- AI 요약 후보, 보고서 문구 및 Export 표시값은 원본이 아니라 파생 데이터로 취급한다.
- 보고서 또는 Export 생성 시 포함된 값은 생성 시점 snapshot으로 추적할 수 있어야 한다.

## 3. 저장 구조 후보 비교

| 후보 | 구조 | 장점 | 주요 위험·제약 | v1 판단 |
| --- | --- | --- | --- | --- |
| A. 기존 현장참여 DB에 함께 저장 | 현재 일반 근로자 `field participation` 저장소에 제출 유형 또는 구분값을 추가 | 기존 제출 경로와 운영 도구를 일부 재사용하기 쉬움 | 일반 공유확인·제보·조치 KPI와 혼합될 위험, 대표 원문 및 관리자 메모 권한 분리 어려움, 정정·검토 이력 확장 시 구조 복잡 | **독립 행·독립 조회·독립 집계를 보장하는 임시 물리 저장 외에는 제외**. 기존 일반 제출 행과 혼합하는 방식은 v1 제외안 |
| B. 별도 `worker_representative_confirmations` 원장 생성 | 대표 확인 전용 테이블 또는 동등한 별도 컬렉션에 원본·상태·연결키 저장 | 책임과 권한 경계가 명확하고 중복·정정·감사·보고 연결을 독립 설계 가능 | 초기 스키마·관리 화면·이관 기준 필요 | **v1 논리 구조 권장안** |
| C. Supabase-first 원장 + Notion 백오피스 보조 | Supabase를 원본 원장으로 사용하고 Notion에는 운영상 필요한 최소 참조·요약만 동기화 | 행 수준 권한, 관계, 이력, 조회 및 Export 확장에 유리; Notion 장애·속성 변경과 원본을 분리 가능 | 정책, migration, 동기화 실패·재처리, 운영 전환 계획 필요 | **목표 저장 구조 권장안**. 준비가 완료된 구현 단계에서 채택 |

### 3.1 v1 권장안

v1은 다음 조합을 권장한다.

1. 제품 모델에서는 별도 `worker_representative_confirmations` 원장을 기준으로 설계한다.
2. 초기 구현은 현재 앱 구조에 맞는 최소 저장 경로를 선택할 수 있다.
3. 최소 경로를 사용하더라도 일반 현장참여 기록과 독립된 행, 조회, 권한, 상태 및 집계를 유지한다.
4. 저장 인터페이스와 식별자를 특정 백오피스 표현에 종속시키지 않아 Supabase-first 전환 가능성을 열어둔다.
5. 장기 목표는 Supabase를 원본 원장으로 두고 Notion은 백오피스 보조 조회·운영 큐로 제한하는 구조다.

초기 물리 저장소의 최종 선택은 후속 submit route PR에서 현재 배포 환경, 고객사별 설정, 권한 정책 및 migration 준비도를 확인한 뒤 결정한다. 이 문서는 특정 저장 제품의 즉시 도입을 확정하지 않지만, 논리 원장 분리는 확정한다.

### 3.2 v1 제외안

다음 방식은 v1에서 제외한다.

- 일반 공유확인 행에 대표 필드만 추가하고 같은 목록·집계로 처리하는 방식
- 일반 근로자 `field participation` KPI 또는 관리자 조치완료 KPI에 대표 확인을 합산하는 방식
- Notion 페이지 URL을 원장 식별자 또는 고객용 증빙 식별자로 직접 사용하는 방식
- Notion과 Supabase를 동시에 독립 원본으로 간주하는 양방향 무주체 저장 방식
- 클라이언트가 보낸 회사 코드, 위험성평가 ID 또는 검토 상태를 서버 검증 없이 신뢰하는 방식
- 정정 시 기존 원문을 수정하거나 삭제해 이전 상태를 복원할 수 없게 하는 방식
- 첨부의 공개 직접 URL을 원장과 보고서·Export에 영구 저장하는 방식

## 4. v1 권장 논리 원장 구조

### 4.1 원장 단위

대표 확인 1건은 특정 회사·현장과 위험성평가 범위에 연결된 독립 기록이다. 최소 논리 단위는 다음 그룹으로 나눈다.

| 그룹 | 주요 필드 후보 | 저장 원칙 |
| --- | --- | --- |
| 식별·연결 | `confirmationId`, `relatedCompanyCode`, `siteId`, `riskAssessmentId`, `confirmationScope` | URL 표시값이 아닌 서버 검증된 연결값 저장 |
| 대표 제출 원본 | `representativeName`, `representativeDepartment`, `representativeRole`, `confirmedAt`, `opinion`, `hasObjection`, `objectionDetail`, `consentChecked` | 입력 원문과 제출 시각 보존, 최소수집 적용 |
| 첨부 참조 | 첨부 ID, 파일명, 유형, 크기, 검사 상태, 비공개 저장 참조 | 직접 공개 URL 대신 권한 검사를 거치는 참조 사용 |
| 관리자 검토 | `reviewStatus`, `reviewedAt`, `reviewedBy`, `reviewNote` | 대표 원문과 분리, 관리자 권한으로만 변경 |
| 생명주기 | `createdAt`, `createdBy`, `supersedesId`, `correctionReason`, `withdrawnAt` | 정정·철회 시 원본 보존과 관계 기록 |
| 출력 연결 | `monthlyReportIncluded`, `exportIncluded`, 생성물 ID, 포함 시점 snapshot | 현재값만 보지 않고 어떤 출력에 무엇이 포함됐는지 추적 |
| 감사 이력 | 이벤트 ID, 대상 ID, 행위, 이전·이후 상태, 수행 주체, 시각, 사유 | 민감 원문 전체 복제를 피하고 변경 사실과 책임 경계 기록 |

필드의 상세 타입, 필수 여부 및 화면별 노출은 `SAFEMETRICA_WORKER_REPRESENTATIVE_CONFIRMATION_FIELDS_V1.md`를 우선한다.

### 4.2 원장과 이력의 관계

- 대표 확인 원장은 현재 유효 상태를 조회하기 위한 기본 레코드를 가진다.
- 상태 변경, 정정, 철회, 보고서 포함 및 Export 생성은 append-only 이벤트 또는 동등한 감사 이력으로 남긴다.
- 감사 이력은 비밀번호, 토큰, 환경변수 값 또는 첨부 다운로드 주소를 저장하지 않는다.
- 로그에 대표 의견·이견 원문 전체를 반복 복제하지 않는다.
- 물리 삭제가 필요한 개인정보 정책 상황에서도 법적·계약상 보존 기준과 비식별 감사정보의 처리 방식을 별도 검토한다.

### 4.3 Notion 백오피스 보조 기준

Supabase-first 전환 후 Notion을 사용한다면 다음 범위로 제한한다.

- 관리자 운영 큐에 필요한 표시명, 상태, 확인일, 이견 여부 및 앱 상세 링크 후보
- 동기화 시각, 동기화 상태 및 재처리 필요 여부
- 원본 원장의 안정적인 외부 참조 ID

Notion에는 다음 정보를 기본 복제하지 않는다.

- 대표 의견·이견 원문 전체
- 관리자 내부 메모 전체
- 첨부 직접 주소
- 인증정보, 세션정보 또는 환경설정 값
- Owner 전용 내부 링크와 원본 payload

Notion 동기화 실패는 원본 제출 실패로 재해석하지 않는다. 원본 원장 저장 성공과 백오피스 보조 동기화 상태를 분리하고 재처리 가능하게 한다.

## 5. 접근 경로 후보

| 경로 후보 | 주요 사용자·목적 | v1 판단 | 권한·연결 주의사항 |
| --- | --- | --- | --- |
| `/field/representative-confirmation?company=xxx` | QR 또는 회사별 전용 링크에서 대표 확인 시작 | 범용 진입 후보 | 회사 코드는 화면 초기화 힌트이며 서버에서 유효성·활성 상태 재검증 |
| `/risk/share/representative?company=xxx&riskId=xxx` | 특정 위험성평가 공유 화면에서 대표 확인 시작 | 위험성평가 직접 연결 후보 | `riskId`의 회사·현장 소속, 공개 가능 버전 및 링크 만료 검증 필요 |
| `/manager/representative-confirmations` | 관리자 검토 목록, 상세, 상태 변경 및 정정 관계 조회 | 관리자 inbox 권장 후보 | 관리자/대표 권한, tenant 경계 및 감사 이력 필요 |
| `/monthly-report` 내 별도 섹션 | 대상 월의 대표 확인 상태와 이견 검토 현황 요약 | 보고서 연결 후보 | 일반 공유확인 지표와 분리, 원문 최소노출 |
| `/owner/export` 내 고객용 Export | 권한 있는 운영자가 고객 전달용 대표 확인 출력 생성 | Export 준비 후보 | Owner 내부 정보 제거, 고객·기간·회사 범위 재검증 |

### 5.1 대표 입력 경로 선택 원칙

두 입력 경로는 서로 배타적이지 않다.

- 회사·현장 단위 QR에는 `/field/representative-confirmation` 후보를 사용할 수 있다.
- 특정 위험성평가에서 시작하는 링크에는 `/risk/share/representative` 후보가 연결 관계를 더 명확히 제공한다.
- 어느 경로든 제출 전에 대상 회사, 현장, 위험성평가 제목·버전·범위를 사람이 확인할 수 있게 표시한다.
- query string만으로 제출 권한을 부여하지 않는다.
- 전용 링크가 필요한 경우 만료, 재사용, 폐기 및 잘못된 tenant 접근 차단 기준을 후속 구현에서 정의한다.

### 5.2 관리자·Owner 경로 분리

- 일상 검토는 `/manager/representative-confirmations`에서 수행하는 구조를 우선한다.
- `/owner/export`는 고객용 산출물 생성과 운영 지원을 위한 별도 권한 경로이며, 일상 검토 상태 변경 화면을 대체하지 않는다.
- Owner Console의 내부 식별자, 운영 링크, 토큰 및 환경변수 값은 대표 입력 화면, 관리자 일반 화면, 월간보고서 및 고객용 Export에 노출하지 않는다.

## 6. 권한 기준

| 역할 | 허용 후보 | 금지·제한 |

### 6.2 서버 권한 검증

- 회사 코드, 위험성평가 ID, 현장 ID 및 역할은 클라이언트 입력만으로 확정하지 않는다.
- 관리자 목록과 상태 변경은 인증된 사용자의 tenant 및 역할을 서버에서 검증한다.
- 대표 입력 링크는 공개 URL 형태일 수 있어도 대상 범위, 링크 유효성 및 제출 허용 조건을 서버에서 검증한다.
- 상태 변경, 정정, 보고서 포함 및 Export 포함은 수행 주체와 시각을 감사 이력에 남긴다.
- 첨부 열람은 원장 상세 권한과 별도로 검사하고, 필요 시 짧은 만료시간의 다운로드 경로를 사용한다.

### 6.3 노출 금지 정보

다음 정보는 일반 근로자 화면과 고객용 기본 산출물에 노출하지 않는다.

- 대표 의견·이견 원문과 관리자 내부 메모
- 다른 회사·현장의 대표 이름 또는 기록
- 내부 DB ID, 백오피스 원본 링크 및 raw payload
- Owner Console 링크·세션정보
- 토큰, 인증정보 및 환경변수 값
- 첨부 저장소의 직접 주소

## 7. v1 데이터 흐름

### 7.1 기본 흐름

```text
대표 확인 제출
→ 입력값·대상 범위·링크 유효성 서버 검증
→ 근로자대표 참여확인 별도 논리 원장 저장
→ 제출 및 생성 감사 이력 기록
→ 관리자 검토 목록 노출
→ reviewStatus 관리 및 상태 변경 이력 기록
→ 월간보고서 별도 섹션 포함 후보 검토
→ 고객용 Export 별도 시트/섹션 포함 후보 검토
→ 생성 시점 snapshot 및 생성 이력 기록
```

### 7.2 단계별 책임 경계

| 단계 | 입력 | 처리 | 출력·다음 단계 |
| --- | --- | --- | --- |
| 1. 대표 확인 제출 | 대상 범위, 대표 정보, 확인일, 의견, 이견, 동의, 첨부 후보 | 필수값, 회사·위험성평가 연결, 파일 및 링크 조건 검증 | 검증된 제출 payload |
| 2. 원장 저장 | 검증된 payload | 고유 ID 부여, 원본 저장, 초기 `reviewStatus` 설정, 생성 이력 기록 | 독립 대표 확인 기록 |
| 3. 관리자 inbox | 회사·기간·상태 필터 | 권한 범위 내 목록 및 상세 제공 | 검토 대상 선택 |
| 4. 검토 상태 관리 | 상태, 검토자, 시각, 메모 | 허용된 상태 전이 검증, 변경 이력 append | 현재 상태와 변경 이력 |
| 5. 월간보고서 후보 | 대상 기간의 유효 기록 | 별도 섹션 집계, 개인정보 최소화, 이견 상태 표시 | 관리자 확인용 보고 후보 |
| 6. 고객용 Export 후보 | 승인된 회사·기간·포함 기록 | 일반 공유확인과 분리, 마스킹, 첨부 Manifest 처리 | 고객 전달 전 검토 파일 |

### 7.3 실패와 재처리

- 원장 저장 실패 시 성공 화면이나 관리자 목록 반영을 먼저 확정하지 않는다.
- 원장 저장 성공 후 백오피스 동기화가 실패하면 동기화 실패 상태를 별도로 기록하고 원본을 재저장하지 않는다.
- 월간보고서 또는 Export 생성 실패는 원본 확인 기록과 `reviewStatus`를 변경하지 않는다.
- 재시도는 동일 제출의 중복 레코드 생성을 막을 수 있는 요청 식별자 또는 idempotency 후보를 사용한다.
- 부분 실패와 재처리 결과는 운영자가 확인할 수 있어야 하며 고객 산출물에는 내부 오류 상세를 포함하지 않는다.

## 8. 중복·정정·이견 처리

### 8.1 중복 후보

같은 `riskAssessmentId + representativeName + confirmedAt` 범위에서 동일하거나 매우 유사한 제출이 발생하면 중복 후보로 표시한다.

- `confirmedAt` 비교 범위는 사업장 기준 시간대와 구현상 허용 오차를 후속 PR에서 확정한다.
- 이름 문자열만으로 동일인을 확정하지 않는다. 회사·현장, 확인 범위, 대표 역할, 제출 요청 식별자 등을 함께 비교한다.
- 네트워크 재시도로 동일 payload가 반복되면 새 기록 생성보다 기존 처리 결과 반환을 우선 검토한다.
- 내용 또는 확인 범위가 다르면 자동 삭제·병합하지 않고 관리자 검토 대상으로 둔다.
- 중복 후보 판정은 운영 신호이며 대표 자격이나 제출 진위를 확정하는 판단이 아니다.

### 8.2 정정·철회

정정 시 원본을 덮어쓰지 않는다.

1. 기존 기록은 생성 당시 원본과 이력을 보존한다.
2. 정정 기록에 새 ID를 부여하고 `supersedesId` 등으로 원본과 연결한다.
3. 정정 사유, 요청 주체, 처리 주체 및 시각을 남긴다.
4. 현재 유효본과 과거본을 관리자 화면에서 구분한다.
5. 월간보고서와 Export는 생성 시점에 사용한 버전과 이후 정정 여부를 추적할 수 있어야 한다.
6. 철회는 물리 삭제와 동일하게 처리하지 않고 철회 상태·사유·시각을 우선 기록한다. 실제 삭제 기준은 개인정보 및 보존정책에서 별도 결정한다.

### 8.3 이견 처리

- `hasObjection=true`이면 자동으로 관리자 검토대상에 포함한다.
- 초기 `reviewStatus`는 상위 필드 문서의 상태 기준을 따르고, 이견 있음 기록이 관리자 확인 없이 보고 완료 상태로 이동하지 않게 한다.
- 이견 원문과 첨부, 검토 상태, 검토자, 검토 메모 및 별도 조치 기록의 연결을 구분한다.
- 이견이 있다는 사실을 일반 공유확인 수나 조치 KPI로 상쇄하지 않는다.
- 별도 위험제보 또는 개선조치가 필요하면 관리자가 원본 확인 ID를 참조하는 별도 기록을 만든다.

### 8.4 AI 사용 범위

- AI는 의견·이견 요약, 민감정보 검토 알림, 유사 제출 또는 분류의 **후보**만 제안할 수 있다.
- AI 출력은 원문과 별도 필드 또는 별도 파생 레코드에 저장하고 생성 시각·모델/버전 후보와 관리자 채택 여부를 구분한다.
- 관리자가 원문과 첨부를 확인한 뒤 요약의 채택·수정·폐기를 결정한다.
- AI는 대표 자격, 중복 확정, 이견 처리 결과, 조치완료 또는 위험성평가 결과를 결정하지 않는다.

## 9. 월간보고서 연결

### 9.1 별도 섹션 원칙

월간보고서에는 `근로자대표 참여확인` 별도 섹션을 둔다. 일반 공유확인, 위험제보, 아차사고, 개선제안 및 관리자 조치 KPI와 같은 수치로 합산하지 않는다.

표시 후보는 다음과 같다.

- 대상 회사·현장·기간
- 연결된 위험성평가 제목 또는 표시용 식별정보
- 확인 상태와 확인일
- 확인 범위
- 이견 있음/없음/미입력
- 관리자 `reviewStatus`
- 다음 확인 필요 여부
- 정정본 또는 철회 기록 존재 여부

### 9.2 개인정보 최소노출

- 기본 월간 요약에는 대표 이름, 소속, 직책 및 의견·이견 원문을 원칙적으로 제외한다.
- 업무상 필요한 권한 있는 상세본만 최소 범위로 표시하고, 수신자와 전달 목적을 확인한다.
- 관리자 내부 메모와 AI 요약 초안은 고객용 보고서에 자동 포함하지 않는다.
- 첨부는 개수·존재 여부 또는 안전한 Manifest만 표시하고 직접 URL은 넣지 않는다.
- 데이터가 없거나 연결되지 않았으면 임의의 확인 상태나 의견을 생성하지 않는다.

### 9.3 생성 이력

보고서 생성 시 다음 snapshot 후보를 남긴다.

- 보고서 ID, 회사·현장, 대상 기간 및 기준 시간대
- 포함된 대표 확인 ID와 당시 유효 버전
- 당시 `reviewStatus`, 이견 여부 및 정정 상태
- 생성 주체와 생성 시각
- 개인정보 마스킹 및 상세 포함 옵션

보고서 재생성은 기존 생성 이력을 덮어쓰지 않고 새 생성 버전으로 추적한다.

## 10. 고객용 Export 연결

### 10.1 일반 공유확인과 분리

고객용 Export에는 일반 공유확인 목록과 분리된 `근로자대표 참여확인` 시트, 섹션 또는 별도 파일을 사용한다. 단일 CSV 형식만 가능한 경우에도 레코드 유형, 컬럼 계약 및 파일명을 분리해 두 기록을 같은 행 구조로 혼합하지 않는다.

대표 확인 Export 컬럼 후보는 다음과 같다.

- 고객 표시용 확인 ID
- 회사·현장
- 위험성평가 표시용 식별정보와 확인 범위
- 확인일
- 대표 역할 또는 소속의 최소 표시값
- 이견 있음 여부
- 관리자 검토 상태와 검토일
- 정정·철회 여부
- 첨부 존재 여부 또는 Manifest 참조

대표 이름과 의견·이견 원문은 기본 Export에서 제외하고, 명확한 업무 목적과 권한이 확인된 별도 상세 Export에서만 최소 범위 포함을 검토한다.

### 10.2 제외 정보

고객용 Export에 다음 정보를 포함하지 않는다.

- 첨부 직접 URL 또는 저장소 경로
- 관리자 내부 메모
- AI 원시 응답과 내부 분류 신뢰도
- 내부 DB UUID, raw payload 및 백오피스 원본 링크
- Owner Console 링크와 내부 운영 계정정보
- 토큰, 인증정보 및 환경변수 값
- 다른 회사·현장의 기록

### 10.3 Export 생성 통제

- Export 실행 전 회사·현장·기간·수신 목적·포함 범위를 다시 확인한다.
- `exportIncluded`는 검토용 플래그이며 고객 전달 승인 또는 원문 공개 동의로 자동 해석하지 않는다.
- 생성 파일에는 생성 시각, 기준 기간 및 데이터 기준을 표시한다.
- 다운로드 권한, 만료, 재다운로드, 전달 및 파기 기준은 후속 구현에서 정한다.
- Export 생성 이력에는 생성 주체, 필터, 포함 ID, 마스킹 옵션 및 결과 상태를 남긴다.

## 11. 금지 표현

화면, 관리자 inbox, 월간보고서, 고객용 Export, 제안서 및 운영 안내에서 다음 표현을 사용하지 않는다.

- `법적 완료`
- `산안법 의무 충족 보장`
- `과태료 방지`
- `법적 면책`
- `중대재해 면책`
- `무재해 보장`
- `근로자대표 확인만으로 위험성평가 완료`
- `AI가 이견 해소 판단`

대신 실제 데이터와 처리 상태에 맞춰 `대표 확인 기록`, `관리자 검토대상`, `검토 중`, `이견 있음`, `자료 없음`, `연결 전`, `운영기록 후보`처럼 제한된 표현을 사용한다.

## 13. 후속 구현 PR

다음 작업은 각각 별도 PR로 진행한다.

1. `feat: add worker representative confirmation submit route`
   - 서버 검증, 원장 저장 인터페이스, 중복·재시도 및 감사 이벤트를 구현한다.
2. `feat: add worker representative confirmation form`
   - QR/전용 링크 진입, 대상 범위 표시, 대표 입력, 이견 및 동의 흐름을 구현한다.
3. `feat: add manager representative confirmation inbox`
   - tenant 권한 목록, 상세, `reviewStatus`, 정정 관계 및 관리자 검토 이력을 구현한다.
4. `feat: show worker representative confirmation in monthly report`
   - 일반 공유확인 KPI와 분리된 섹션, 최소노출 및 생성 snapshot을 구현한다.
5. `feat: add customer export columns for representative confirmation`
   - 별도 시트·섹션, 마스킹, 첨부 Manifest 및 Export 생성 이력을 구현한다.

각 구현 PR은 이 문서와 상위 필드 문서의 기준을 함께 확인하고, 실제 저장소·권한·개인정보 정책 변경이 필요하면 migration 및 롤백 계획을 포함한다.
