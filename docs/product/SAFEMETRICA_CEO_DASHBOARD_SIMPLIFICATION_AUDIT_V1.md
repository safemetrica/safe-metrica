# SafeMetrica 대표 대시보드 단순화 정보구조 감사 V1

- 작업명: `docs: audit ceo dashboard simplification v1`
- 문서 성격: 코드 수정 전 정보구조(IA) 감사
- 감사 대상: `/dashboard` 대표 대시보드
- 핵심 판단: 현재 대표 대시보드는 정보가 부족한 화면이 아니라, 대표가 첫 판단을 내리기 전에 너무 많은 정보와 역할별 상세를 동시에 읽어야 하는 화면이다.

---

## 1. 감사 목적

대표 대시보드의 첫 화면은 기능 모음판이나 운영 데이터 총람이 아니라 다음 질문에 **10초 안에 답하는 화면**이어야 한다.

> 오늘 대표가 개입해야 하는가?

이를 위해 첫 화면의 정보 우선순위를 다음 순서로 재정의한다.

1. 오늘 확인하거나 개입할 일이 있는가
2. AI 운영브리핑이 요약한 핵심 상황은 무엇인가
3. 대표가 직접 확인할 Risk 신호가 있는가
4. 더 자세한 정보는 어디에서 확인하는가

이번 감사는 삭제할 기능을 결정하는 작업이 아니다. 대표의 첫 판단에 필요한 정보와 현장 운영 상세를 분리하고, 상세 정보는 접힘 영역 또는 하위 화면에서 계속 접근할 수 있도록 방향을 정하는 작업이다.

## 2. 변경 금지 및 감사 범위

### 2.1 이번 문서 PR에서 변경하지 않는 것

- AI 운영브리핑의 문구
- AI 운영브리핑의 생성 로직
- AI 운영브리핑의 표시 조건
- `dailySafetyBriefing.ts`
- API, DB, Notion, Supabase 연동
- 조회·저장·상태 변경 로직
- TS/TSX 파일

AI 운영브리핑은 첫 화면에 유지할 핵심 영역으로만 분류하며, 내부 내용과 동작은 후속 구현 PR에서도 별도 요구가 없는 한 그대로 보존한다.

### 2.2 이번 감사가 다루는 것

- 대표 첫 화면의 정보량과 시각적 우선순위
- 서로 중복되는 섹션과 용어
- 대표용 정보와 현장관리자용 상세 정보의 혼재
- 첫 화면 유지, 축소·접힘, 하위 화면 이동 후보
- 후속 구현 PR의 분리 순서

## 3. 현재 대표 대시보드 구조 요약

현재 화면은 크게 네 구역으로 구성되어 있다.

| 현재 구역 | 주요 내용 | 대표 첫 판단과의 관계 |
|---|---|---|
| ZONE 1: 오늘의 결론 | 오늘 확인 요약, 현장 위험도 점수, 고위험 항목 수 | 핵심 판단에 가깝지만 숫자 카드가 동시에 경쟁함 |
| ZONE 2: 핵심 의사결정 | AI 운영브리핑, 오늘 처리할 항목, 위험관리 요약 | 핵심 영역과 현장 처리 상세가 한 구역에 혼재함 |
| ZONE 3: 증빙/운영 상태 | 증거 완결성, 운영 KPI, TBM/EB 진입 | 대표의 즉시 개입 판단보다 운영 점검·빠른 통계 성격이 강함 |
| ZONE 4: 상세 상태 | PTW 상세 리스트, 오늘 할 일 | 상단의 오늘 처리할 항목 및 Risk/PTW 신호와 중복됨 |

현재 렌더링 구조만 보더라도 대표는 첫 화면에서 다음을 연속해서 해석해야 한다.

- 오늘의 결론 문장
- 현장 위험도 점수
- 고위험 항목 수와 조치·비용 수치
- AI 운영브리핑의 헤드라인과 복수 메시지
- 조치 필요, 증빙 누락, PTW 승인 대기, PTW 금지/반려 카드
- 위험관리 역할별 카드와 처리 현황 배지
- 고위험 상세 항목 목록
- 증거 완결성 점수
- 운영 KPI 네 개
- TBM/EB 바로가기
- PTW 상세 목록
- 오늘 할 일 목록

각 정보는 개별적으로 의미가 있지만, 모두 펼쳐진 상태에서는 대표가 무엇을 먼저 봐야 하는지보다 “확인 가능한 기능과 데이터가 얼마나 많은지”가 먼저 드러난다.

## 4. 현재 대표 대시보드의 핵심 문제

### 4.1 정보가 너무 많아 첫 시선의 목적지가 불명확하다

첫 화면에 결론, 점수, 건수, 브리핑, 작업 카드, KPI, 상세 목록이 모두 높은 강조도로 노출된다. 고객은 각 카드의 의미를 이해한 뒤 우선순위를 다시 조합해야 한다.

대표 화면이 제공해야 할 것은 더 많은 숫자가 아니라 다음과 같은 명확한 판단이다.

- `확인 필요 없음`
- `오늘 확인 필요`
- `대표 개입 필요`

현재는 이 판단을 지원하는 신호들이 여러 구역에 분산되어 있다.

### 4.2 오늘 할 일, Risk 신호, KPI, 상세 항목이 한 화면에 과다 노출된다

현재 상단의 `오늘의 결론`, 중단의 `오늘 처리할 항목`, 하단의 `오늘 할 일`이 유사한 운영 이슈를 서로 다른 집계와 표현으로 반복한다. PTW는 오늘 처리 카드와 하단 상세 리스트에서 다시 노출되고, 고위험 Risk는 상단 숫자 카드와 위험관리 요약 및 상세 항목으로 반복된다.

그 결과 첫 화면에서 다음 층위가 분리되지 않는다.

- 대표가 오늘 개입 여부를 결정하는 신호
- 담당자가 실제로 처리해야 하는 업무
- 운영 현황을 설명하는 KPI
- 원본 항목에 가까운 상세 리스트

### 4.3 대표용 화면에 현장관리자용 상세 정보가 섞여 있다

위험관리 요약은 대표, 현장감독자, 담당자가 함께 확인하는 구조이고 역할별 처리 단계와 개별 고위험 항목까지 보여 준다. PTW 제목·작업일·작업유형·승인상태 목록과 TBM/EB 운영 수치도 현장 실행과 점검에 유용하지만 대표의 첫 10초 판단에는 과도한 상세다.

대표에게 필요한 것은 모든 처리 단계가 아니라 다음과 같은 예외 신호다.

- 중대한 Risk가 새로 발생했는가
- 조치 지연이나 승인 병목이 대표 개입 수준인가
- 비용 또는 의사결정이 필요한가
- 상세 확인이 필요할 때 어느 화면으로 이동해야 하는가

### 4.4 카드 수와 강조 수준이 우선순위를 약화한다

여러 카드가 유사한 테두리, 큰 숫자, 상태색을 사용한다. 카드가 많아질수록 색상은 중요도를 전달하기보다 구역을 구분하는 장식으로 작동할 수 있다. 특히 모바일에서는 세로 스크롤 중 핵심 결론과 하위 운영 정보의 위계가 더 약해진다.

### 4.5 기간이 다른 지표가 같은 판단 흐름에 섞인다

오늘의 결론과 오늘 처리 항목 사이에 전체 누적, 이번 달 TBM, 이번 달 특이사항 같은 기간 지표가 같은 화면에 놓인다. 월간보고서 성격의 그래프나 추세가 추가되거나 노출될 경우에도 오늘의 개입 판단과 분리해야 한다.

## 5. 첫 화면에 남길 영역

첫 화면의 기본 노출 영역은 다음 네 가지로 제한한다. 각 영역은 하나의 질문에만 답해야 한다.

### 5.1 오늘 확인 필요

**답할 질문:** 오늘 대표가 확인하거나 개입할 항목이 있는가?

- 첫 화면의 최상위 결론으로 유지한다.
- 여러 운영 수치를 나열하기보다 상태와 총량을 먼저 보여 준다.
- 필요한 경우 가장 중요한 1~3개 항목만 요약한다.
- 세부 조치 대상은 상세 화면으로 이동한다.
- `오늘 할 일`과 `오늘 처리할 항목`은 이 영역의 중복 후보이므로 하나의 명칭과 집계 원칙으로 통합한다.

권장 정보 순서:

1. 상태: 안정 / 확인 필요 / 대표 개입 필요
2. 확인 필요 총 건수
3. 가장 중요한 사유
4. 상세 보기 동작

### 5.2 AI 운영브리핑

**답할 질문:** 오늘 운영 상황을 한 문장으로 어떻게 이해해야 하는가?

- 첫 화면에 그대로 유지한다.
- 이번 감사와 후속 단순화 구현에서 브리핑 문구, 생성 로직, 표시 조건을 수정하지 않는다.
- 다른 카드가 브리핑의 결론을 반복하지 않도록 주변 정보구조만 정리한다.
- AI 운영브리핑은 독립된 핵심 영역이며 `오늘 확인 필요`의 대체물이 아니다.

### 5.3 대표 Risk 확인 신호

**답할 질문:** 대표 판단 또는 자원 투입이 필요한 Risk가 있는가?

- `오늘 할 일`과 분리된 독립 카드로 유지한다.
- 일반 담당자 처리량보다 대표 의사결정 신호를 우선한다.
- 우선 후보는 고위험, 비용 검토, 장기 미조치, 승인 병목 등이다.
- 첫 화면에서는 대표가 확인할 신호의 수와 가장 중요한 사유까지만 보여 준다.
- 위험성평가 전체 처리 현황과 개별 항목 목록은 Risk 상세 화면으로 이동한다.

### 5.4 상세 메뉴 진입

**답할 질문:** 더 자세히 보려면 어디로 이동하는가?

- TBM, EB, PTW, Risk 등 상세 기능은 버튼 또는 하위 화면 진입점으로 제공한다.
- 상세 메뉴는 데이터 카드처럼 경쟁하지 않도록 시각적 강조를 낮춘다.
- 대표 첫 화면에 상세 리스트를 펼쳐 놓는 대신 필요한 메뉴를 명확히 찾을 수 있게 한다.
- 모바일에서는 하나의 `상세 메뉴` 진입점 또는 최소한의 바로가기만 우선 노출한다.

## 6. 첫 화면에서 줄이거나 접을 영역

| 영역 | 현재 문제 | 권장 처리 |
|---|---|---|
| 운영 KPI | 전체·이번 달·특이사항·누락 수치가 오늘 판단과 같은 강조도로 노출됨 | 기본 접힘 또는 운영 현황 하위 화면으로 이동 |
| 빠른 통계 | 여러 숫자가 결론보다 먼저 비교를 요구함 | 핵심 상태에 포함되지 않는 통계는 접힘 처리 |
| 상세 리스트 | PTW 및 고위험 개별 항목이 첫 화면을 길게 만듦 | 대표 신호 1~3개만 요약하고 전체는 상세 화면으로 이동 |
| 위험성평가 전체표/전체 현황 | 역할별 처리 단계와 전체 상태는 현장관리 성격이 강함 | 대표 Risk 신호만 남기고 전체표·처리 현황은 `/risk`로 이동 |
| 중복된 오늘 할 일/오늘 처리할 항목 | 유사 이슈가 서로 다른 섹션과 건수로 반복됨 | 하나의 `오늘 확인 필요` 영역으로 통합 |
| 월간보고서 성격의 그래프와 추세 | 오늘 개입 여부와 시간 범위가 다름 | 첫 화면 기본 노출에서 제외하고 월간보고서 또는 접힘 영역으로 이동 |
| 증거 완결성 | 운영 품질 점검에는 유용하지만 즉시 개입 여부와 직접 연결되지 않을 수 있음 | 이상 신호만 오늘 확인 필요에 반영하고 점수·분석은 접힘 처리 검토 |
| PTW 현황 상세 | 오늘 처리 카드 및 Risk 신호와 중복 가능 | 예외 건수만 요약하고 제목·작업정보 목록은 `/ptw`로 이동 |
| TBM/EB 개별 바로가기 카드 | KPI와 함께 배치되어 별도 정보 카드처럼 보임 | 상세 메뉴 진입 영역으로 통합 |

`줄이거나 접는다`는 데이터나 기능을 제거한다는 뜻이 아니다. 대표 첫 화면의 기본 노출에서 우선순위를 낮추고, 사용자가 필요할 때 접근하도록 정보 계층을 조정한다는 뜻이다.

## 7. 용어 정리

### 7.1 `오늘 할 일`과 `오늘 처리할 항목`

두 용어가 같은 화면에 함께 있으면 사용자는 차이를 추론해야 한다. 실제 집계 대상이 일부 다르더라도 대표 관점에서는 모두 오늘 확인해야 할 운영 예외로 인식될 가능성이 높다.

**권장 대표 화면 용어: `오늘 확인 필요`**

- 대표에게 직접 작업 수행을 요구하는 표현보다 확인·개입 판단에 적합하다.
- 담당자의 실행 업무는 상세 화면에서 `처리할 항목` 또는 기존 업무 용어를 사용할 수 있다.
- 통합 시 건수 산정 규칙과 중복 제거 규칙을 후속 구현 PR에서 명시해야 한다.

### 7.2 `대표 Risk 확인 신호`

- `오늘 확인 필요`와 분리한다.
- 일반 운영 누락과 대표 의사결정 Risk를 같은 할 일 목록에 섞지 않는다.
- 대표 Risk 카드는 고위험·비용·장기 지연·승인 병목처럼 대표 역할과 연결되는 기준을 가져야 한다.
- 단순히 위험성평가 전체 항목 수를 다시 보여 주는 카드가 되어서는 안 된다.

### 7.3 상세 기능 명칭과 이동 원칙

- 상세 기능은 첫 화면의 독립 데이터 섹션보다 버튼 또는 하위 화면으로 이동한다.
- 버튼명은 목적지가 명확해야 한다. 예: `Risk 상세 보기`, `PTW 상세 보기`, `운영 KPI 보기`.
- 첫 화면의 요약 수치와 상세 화면의 원본 목록 사이에 의미가 달라지지 않도록 동일한 집계 정의를 사용한다.

### 7.4 권장 용어 체계

| 정보 층위 | 권장 용어 | 의미 |
|---|---|---|
| 최상위 운영 판단 | 오늘 확인 필요 | 오늘 대표가 확인할 운영 예외 |
| AI 요약 | AI 운영브리핑 | 기존 브리핑을 변경 없이 유지 |
| 대표 의사결정 | 대표 Risk 확인 신호 | 대표 개입·의사결정 가능성이 있는 Risk |
| 세부 실행 | 처리할 항목 | 담당자 또는 현장관리자의 실제 처리 업무 |
| 원본/전체 정보 | 상세 보기 | 하위 화면으로 이동하는 동작 |

## 8. 목표 정보구조 V1

### 8.1 데스크톱 기본 노출

1. **오늘 확인 필요**
   - 상태 문장
   - 확인 필요 총량
   - 우선 사유 최대 3개
   - 상세 보기
2. **AI 운영브리핑**
   - 현재 문구·로직·조건 그대로 유지
3. **대표 Risk 확인 신호**
   - 대표 판단 대상 수
   - 핵심 사유
   - Risk 상세 보기
4. **상세 메뉴 진입**
   - TBM / EB / PTW / Risk / 운영 현황
5. **접힘 영역**
   - 운영 KPI
   - 빠른 통계
   - 증거 완결성
   - 월간 그래프·추세

### 8.2 모바일 기본 노출

모바일 첫 화면은 **4카드 이하**를 원칙으로 한다.

권장 순서:

1. 오늘 확인 필요
2. AI 운영브리핑
3. 대표 Risk 확인 신호
4. 상세 메뉴

운영 KPI, 빠른 통계, 상세 리스트, 월간 추세는 초기 화면에서 펼치지 않는다. 사용자가 스크롤을 시작하기 전에 오늘 개입 여부를 판단할 수 있어야 한다.

### 8.3 상태별 노출 원칙

- 문제가 없을 때도 빈 상세 카드 여러 개를 보여 주지 않고 안정 상태를 짧게 전달한다.
- 문제가 있을 때는 모든 원본 항목을 나열하지 않고 우선순위가 높은 사유를 제한적으로 보여 준다.
- 대표 Risk가 없으면 카드 내부를 복잡한 0 지표로 채우지 않고 `대표 확인이 필요한 Risk 없음`으로 명확히 표현한다.
- 상세 데이터 접근 경로는 상태와 관계없이 유지한다.

| 월간 그래프·추세 | 월간보고서/접힘 | 첫 화면 기본 노출 제외 |

## 10. 후속 구현 PR 방향

### PR 1: 중복 섹션 제거 및 첫 화면 카드 축소

- `오늘 할 일`과 `오늘 처리할 항목`의 중복을 제거한다.
- 대표 화면의 통합 명칭을 `오늘 확인 필요`로 정리한다.
- 상단 결론과 중복되는 숫자·설명 카드를 축소한다.
- 데이터 생성·저장 로직은 변경하지 않는다.
- AI 운영브리핑은 변경하지 않는다.

### PR 2: 운영 KPI/빠른 통계 접힘 처리

- 운영 KPI와 빠른 통계를 기본 접힘 상태로 이동한다.
- 증거 완결성과 월간 성격 지표도 같은 운영 정보 계층으로 묶을지 검토한다.
- TBM/EB 진입은 상세 메뉴로 유지한다.
- 접힘 상태에서도 접근 가능성과 키보드 사용성을 보장한다.

### PR 3: 대표 Risk 확인 신호 별도 카드 정리

- 대표 Risk와 일반 오늘 할 일을 분리한다.
- 고위험, 비용 검토, 장기 미조치, 승인 병목 중 대표 신호 기준을 확정한다.
- 역할별 처리 현황과 전체 상세 항목은 Risk 상세 화면으로 이동한다.
- 첫 화면에는 대표가 판단할 최소 요약만 남긴다.

### PR 4: 모바일 첫 화면 4카드 이하로 단순화

- 모바일 기본 노출을 오늘 확인 필요, AI 운영브리핑, 대표 Risk 확인 신호, 상세 메뉴의 최대 4카드로 제한한다.
- 상세 리스트와 통계는 기본 접힘 또는 하위 화면으로 이동한다.
- 주요 결론이 첫 뷰포트 또는 최소 스크롤 내에서 읽히는지 실제 화면으로 검증한다.

## 12. 결론

대표 대시보드의 개선 목표는 정보를 더 추가하는 것이 아니라, 이미 존재하는 정보를 대표의 의사결정 순서에 맞게 줄이고 분리하는 것이다.

첫 화면은 다음 네 가지에 집중해야 한다.

1. 오늘 확인 필요
2. AI 운영브리핑
3. 대표 Risk 확인 신호
4. 상세 메뉴 진입

운영 KPI, 빠른 통계, 상세 리스트, 위험성평가 전체 현황, 중복된 할 일, 월간보고서 성격의 그래프와 추세는 삭제 대상이 아니라 **기본 노출 우선순위를 낮출 대상**이다. 이 원칙을 기준으로 후속 구현을 네 개의 작은 PR로 나누면 AI 운영브리핑과 데이터 로직을 건드리지 않고도 대표 화면의 복잡도를 단계적으로 줄일 수 있다.
