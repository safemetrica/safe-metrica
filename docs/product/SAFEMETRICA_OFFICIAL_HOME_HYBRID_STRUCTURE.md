# SafeMetrica Official App Home Hybrid Structure

기준일: 2026-06-07

문서 목적: SafeMetrica 정식앱 홈을 `역할별 오늘 할 일 + 전체 기능 메뉴` 혼합형 구조로 설계하기 위한 제품 기준을 정의한다.

참조 기준:

* `docs/product/SAFEMETRICA_FEATURE_TRUTH_INVENTORY.md`
* `docs/partner-demo/PARTNER_DEMO_TRUTH_MAP.md`
* `docs/partner-demo/PARTNER_DEMO_UI_SPEC.md`

---

## 1. 목적

정식앱 홈은 고객사의 실제 안전운영 기록을 다루는 시작 화면이다. 사용자가 로그인 직후 자신의 역할에서 오늘 확인하거나 처리해야 할 항목을 먼저 이해하고, 필요할 때 전체 기능으로 이동할 수 있어야 한다.

이 문서는 다음 기준을 고정한다.

* 역할별 오늘 할 일을 1차 내비게이션으로 둔다.
* 기존 전체 기능 메뉴를 2차 내비게이션으로 유지한다.
* 대표, 현장관리자, 근로자에게 서로 다른 우선순위와 문구를 제공한다.
* 현재 기능 상태와 고객사별 사용 범위에 맞춰 메뉴 노출 수준을 구분한다.
* Partner Demo의 샘플 체험 구조와 정식앱의 실제 운영 구조를 명확히 분리한다.
* AI는 후보와 참고정보를 제안하며, 최종 판단과 책임은 관리자와 사업주에게 있음을 유지한다.

이 문서는 정보구조와 제품 표현 기준을 정의한다. 개별 화면의 상세 UI, 데이터 집계식, 권한 정책, 구현 방식은 후속 개발 명세에서 확정한다.

## 2. 현재 문제

### 2.1 기능 중심 홈의 우선순위 부족

전체 기능을 동일한 수준으로 나열하면 사용자는 오늘 해야 할 일과 단순 조회 메뉴를 직접 구분해야 한다. 특히 현장관리자는 TBM 작성, 참여 접수 확인, 미조치 검토처럼 시간 순서가 있는 업무를 놓치기 쉽다.

### 2.2 역할별 목적의 혼재

대표는 주요 운영 신호와 미조치 현황을 빠르게 확인해야 하고, 현장관리자는 접수와 조치를 처리해야 하며, 근로자는 오늘의 위험요인과 안전조치를 확인하고 현장의견을 남겨야 한다. 동일한 홈 구성을 제공하면 각 역할의 핵심 행동이 약해진다.

### 2.3 기능 상태와 메뉴 표현의 불일치 위험

운영중, 제한 운영, 베타, 준비 중, 장기 검토 기능이 같은 방식으로 보이면 사용자가 현재 사용할 수 있는 범위를 오해할 수 있다. 실제 제공 범위가 고객사 설정에 따라 달라지는 기능도 일관된 안내가 필요하다.

### 2.4 Partner Demo와 정식앱의 경험 혼동

Partner Demo는 샘플 데이터로 핵심 흐름을 체험하는 모드이고, 정식앱은 고객사의 실제 운영기록을 저장하고 관리하는 모드다. 두 제품의 역할 구조와 핵심 흐름은 정합성을 유지하되, 저장·권한·안내 문구는 분명히 달라야 한다.

### 2.5 AI 역할의 과대 해석 위험

현장비서와 자동 요약 결과가 확정 판단처럼 보이면 안전조치의 충분성이나 법적 판단을 시스템이 대신하는 것으로 오해될 수 있다. AI 결과는 검토할 후보와 참고정보로 제한하고 관리자 확인 절차를 함께 보여줘야 한다.

## 3. 설계 원칙

### 3.1 오늘 할 일 우선

로그인 직후 첫 화면에는 사용자의 역할과 오늘 상태에 맞는 업무를 우선 노출한다. 카드의 제목은 기능명이 아니라 사용자가 수행할 행동이나 확인할 상태를 중심으로 작성한다.

### 3.2 전체 기능 접근성 유지

역할별 홈으로 전환하더라도 기존 기능을 숨기거나 제거하지 않는다. 전체 기능 메뉴를 2차 영역에 유지해 반복 업무 외의 조회, 기록, 보고서, 설정 기능에도 접근할 수 있게 한다.

### 3.3 역할·고객사·기능 상태의 결합

홈 노출은 다음 세 조건을 함께 반영한다.

1. 로그인 사용자의 역할과 권한
2. 고객사에서 사용하는 모듈과 운영 범위
3. Feature Truth Inventory에 정의된 현재 기능 상태

고객사별 설정의 단일 기준은 후속 구현 명세에서 확정하며, 화면마다 임의 조건을 중복 구현하지 않는다.

### 3.4 운영기록 중심

홈의 수치와 상태는 입력된 운영기록을 요약한다. 확인, 접수, 검토, 조치필요, 조치완료 등 서로 다른 상태를 하나의 완료 지표로 합치지 않는다. 공유확인과 조치성과도 분리한다.

### 3.5 과장 없는 상태 표시

현재 사용할 수 없는 항목은 활성 기능처럼 노출하지 않는다. 제한 운영과 베타는 범위 또는 상태 라벨을 표시하고, 준비 중과 장기 검토는 실행 메뉴가 아닌 안내 또는 로드맵 영역에서만 다룬다.

### 3.6 AI는 후보 제안자

AI 또는 현장비서는 누락 가능성, 요약, 브리핑, 조치 후보를 제안할 수 있다. 결과에는 관리자 검토가 필요하다는 의미를 유지하며, 최종 판단과 책임은 관리자와 사업주에게 있다.

### 3.7 모바일 우선, 행동 중심

홈은 현장에서 짧은 시간 안에 이해하고 실행할 수 있도록 모바일 우선으로 설계한다. 중요한 오늘 할 일은 첫 화면에서 식별할 수 있게 하고, 카드 전체를 명확한 이동 또는 실행 대상으로 구성한다.

## 4. 정식앱 홈 구조

정식앱 홈의 기본 순서는 다음과 같다.

1. 상단: 고객사 / 역할 / 오늘 상태
2. 1차 영역: 역할별 오늘 할 일
3. 2차 영역: 전체 기능 메뉴
4. 하단: 실제 운영 모드 안내

### 4.1 상단: 고객사 / 역할 / 오늘 상태

상단은 현재 사용자가 어느 운영 환경을 보고 있는지 명확히 보여준다.

| 항목 | 표시 기준 | 목적 |
| --- | --- | --- |
| 고객사 | 현재 선택된 고객사 또는 사업장명 | 다른 고객사나 샘플 환경과의 혼동 방지 |
| 역할 | 대표 / 현장관리자 / 근로자 | 역할별 홈과 권한 범위 확인 |
| 기준일 | 오늘 날짜 | 오늘 할 일의 집계 기준 명시 |
| 오늘 상태 | 확인 필요, 처리 필요, 정상 진행 등 기록 기반 요약 | 우선 행동 안내 |
| 사업장 전환 | 권한이 있는 사용자에게만 제공 | 고객사·사업장 범위 유지 |

오늘 상태는 법적 적합성이나 현장 안전의 최종 판정을 뜻하지 않는다. 현재 입력된 운영기록에서 확인 또는 처리가 필요한 항목을 요약하는 용도로 사용한다.

### 4.2 1차 영역: 역할별 오늘 할 일

역할별 오늘 할 일은 홈의 핵심 영역이다.

각 카드는 다음 정보를 기본으로 가진다.

* 행동 또는 상태 중심 제목
* 오늘 기준 건수나 진행 상태
* 기준 시각 또는 최근 업데이트 시각
* 다음 화면에서 수행할 수 있는 행동
* 필요한 경우 `제한 운영`, `베타` 등의 상태 라벨

정렬 우선순위는 다음과 같다.

1. 즉시 확인이 필요한 미조치·조치필요 항목
2. 오늘 작성하거나 확인해야 하는 반복 업무
3. 새로 접수된 현장 의견
4. 운영현황과 보고서 조회
5. 참고 브리핑과 후보 제안

건수가 0인 경우에도 중요한 반복 업무는 숨기지 않고 `확인할 항목 없음`, `작성 전`, `확인 완료` 등 실제 상태를 표시한다. 데이터가 없을 때 임의의 정상 또는 완료 상태를 만들지 않는다.

### 4.3 2차 영역: 전체 기능 메뉴

전체 기능 메뉴는 기존 기능 접근성을 유지하는 2차 내비게이션이다. 메뉴는 기능 목록을 단순히 모두 노출하지 않고 역할 권한, 고객사 사용 모듈, 기능 상태를 반영한다.

권장 그룹은 다음과 같다.

| 그룹 | 포함 가능한 메뉴 |
| --- | --- |
| 오늘 운영 | TBM, 위험성평가 공유확인, 현장참여 |
| 접수·조치 | 위험제보, 아차사고, 개선제안, 현장참여 접수함, 조치상태 관리 |
| 기록·증빙 | 사진첨부, 조치 전후 기록, Evidence Book, PTW |
| 브리핑·지원 | 현장비서, 날씨·폭염 브리핑 |
| 현황·보고 | 대표 대시보드, 위험성평가 현황, 월간보고서, 위험성평가표 출력지원 |
| 설정·관리 | 고객사와 역할 권한에 맞는 운영 설정 |

같은 기능이 1차 영역에 노출되더라도 2차 메뉴에서 제거하지 않는다. 1차 영역은 오늘의 우선 행동으로 연결하고, 2차 영역은 해당 기능의 전체 목록과 이력으로 연결한다.

### 4.4 하단: 실제 운영 모드 안내

정식앱 하단에는 현재 화면이 실제 고객 운영 환경임을 짧고 명확하게 안내한다.

권장 문구:

> 현재 고객사 운영 환경입니다. 입력·확인·조치 기록은 권한과 고객사 운영 기준에 따라 저장·관리됩니다.

필요한 경우 다음 보조 안내를 제공한다.

* 화면에 표시되는 상태는 현재 입력된 운영기록 기준이다.
* AI 브리핑과 제안은 참고정보이며 관리자 확인이 필요하다.
* 최종 판단과 책임은 관리자와 사업주에게 있다.

Partner Demo에서 사용하는 `샘플 데이터`, `샘플 저장`, `실제 고객 DB 미연결` 안내는 정식앱 홈에 사용하지 않는다.

## 5. 역할별 홈 구성

### 5.1 대표

대표 홈은 세부 입력보다 전체 운영 신호, 미조치, 주요 접수, 보고 흐름을 우선한다.

| 우선순위 | 오늘 할 일 카드 | 표시 내용 | 연결 화면 | 기능 상태 기준 |
| --- | --- | --- | --- | --- |
| 1 | 오늘 미조치 | 조치필요·검토중 등 확인이 필요한 건수 | 조치상태 목록 또는 대표 대시보드 | 조치상태 관리 운영중 |
| 2 | 위험제보 접수 | 오늘 신규·미검토 접수 건수 | 위험제보 또는 현장참여 접수함 | 운영중 |
| 3 | TBM 현황 | 오늘 작성·참여·확인 현황 | TBM 현황 | 운영중 |
| 4 | 월간보고서 | 최신 보고 대상 기간과 생성·검토 상태 | 월간보고서 | 베타 라벨 표시 |
| 5 | 위험성평가표 출력지원 | 출력 가능한 기록 범위와 최근 상태 | 위험성평가표 출력지원 | 제한 운영 라벨과 범위 안내 |

대표 홈의 기본 해석 원칙:

* `오늘 미조치`는 현재 기록상 확인 또는 처리가 필요한 항목이며, 전체 안전상태의 최종 판정이 아니다.
* 대표 대시보드의 수치는 역할별 원천 기록으로 이동할 수 있어야 한다.
* 월간보고서는 운영기록 요약이며 관리자와 사업주의 검토를 거친다.
* 위험성평가표 출력지원은 정리와 출력을 지원하는 기능으로 범위를 제한한다.

### 5.2 현장관리자

현장관리자 홈은 오늘의 작성, 접수 검토, 조치, 브리핑 순서로 구성한다. 정식앱 홈을 실제 반영할 때 가장 먼저 구현할 역할이다.

| 우선순위 | 오늘 할 일 카드 | 표시 내용 | 연결 화면 | 기능 상태 기준 |
| --- | --- | --- | --- | --- |
| 1 | 오늘 TBM 작성 | 작성 전·작성 중·작성 완료 상태 | TBM 작성 또는 TBM 현황 | TBM 현황 운영중, 음성작성 베타 |
| 2 | 현장참여 접수함 | 신규 접수, 공유확인, 검토대상 구분 | 현장참여 접수함 | 운영중 |
| 3 | 조치 필요 항목 | 조치필요·검토중·기한 확인 대상 | 조치상태 관리 | 운영중 |
| 4 | 현장비서 | 오늘 확인할 기록과 후보 브리핑 | 현장비서 | 베타 라벨 및 관리자 확인 안내 |
| 5 | 위험성평가 현황 | 공유확인과 관련 운영기록 현황 | 위험성평가 관련 화면 | 고객사 사용 범위에 따라 제한 |
| 6 | 월간보고서 | 현재 월 운영기록 요약과 검토 진입 | 월간보고서 | 베타 라벨 표시 |

현장관리자 홈의 기본 해석 원칙:

* TBM 음성작성 결과는 초안이며 저장 전 관리자가 확인한다.
* 공유확인, 위험제보, 아차사고, 개선제안을 접수함에서 유형별로 구분한다.
* 조치상태와 사진기록은 각각의 상태와 범위를 보여주며 하나의 지표로 충분성을 단정하지 않는다.
* 현장비서는 누락 가능성과 조치 후보를 제안하며 담당자 지정이나 상태 변경은 관리자가 수행한다.
* 위험성평가 관련 기능을 사용하지 않는 고객사에는 빈 화면 대신 사용 범위 안내를 제공한다.

### 5.3 근로자

근로자 홈은 오늘 알아야 할 위험요인과 안전조치를 확인한 뒤, 필요한 현장의견을 쉽게 남기는 흐름으로 구성한다.

| 우선순위 | 오늘 할 일 카드 | 표시 내용 | 연결 화면 | 기능 상태 기준 |
| --- | --- | --- | --- | --- |
| 1 | 오늘 위험요인 확인 | 오늘 작업과 관련해 확인할 위험요인 | 현장참여 위험요인 확인 | 운영중 |
| 2 | 위험성평가 공유확인 | 미확인·확인 완료 상태 | 근로자 위험성평가 공유확인 | 운영중 |
| 3 | TBM 확인 | 오늘 TBM 내용과 확인 상태 | TBM 확인 | 운영중 |
| 4 | 위험제보 / 아차사고 / 개선제안 | 의견 유형 선택과 제출 진입 | 현장참여 제출 화면 | 모두 운영중 |

근로자 홈의 기본 해석 원칙:

* 확인 행동은 근로자가 위험요인과 안전조치를 확인했다는 운영기록을 남기는 흐름이다.
* 공유확인을 조치완료 성과로 합산하지 않는다.
* 제출 항목은 위험제보, 아차사고, 개선제안을 구분해 관리자가 후속 검토할 수 있게 한다.
* 제출 완료는 접수 상태이며 후속 조치의 완료를 의미하지 않는다.
* 근로자에게 관리자 전용 현황, 보고서, 설정 메뉴를 노출하지 않는다.

## 6. 메뉴 노출 기준

모든 메뉴는 Feature Truth Inventory의 상태를 기준으로 다음과 같이 처리한다.

| 상태 | 홈 노출 기준 | 실행 방식 | 필수 표시 |
| --- | --- | --- | --- |
| 운영중 | 역할과 고객사 사용 범위에 맞으면 1차 또는 2차 영역에 노출 | 정상 진입 | 필요 시 오늘 상태와 건수 |
| 제한 운영 | 실제 사용 고객사와 허용 범위에서만 노출 | 활성 진입 또는 범위 안내 | `제한 운영` 라벨, 제공 범위와 한계 |
| 베타 | 실제 작동 범위에서 노출 가능 | 활성 진입 | `베타` 라벨, 검토 필요 안내 |
| 준비 중 | 오늘 할 일과 활성 전체 메뉴에서 제외 | 별도 안내가 필요할 때만 비활성 정보 제공 | `준비 중`, 현재 제공 기능이 아님을 명시 |
| 장기 검토 | 정식앱 실행 메뉴에서 제외 | 제품 로드맵 또는 비전 문서에서만 안내 | `장기 검토`, 미래 방향임을 명시 |

추가 기준:

* `미제공` 기능은 홈과 전체 기능 메뉴에 노출하지 않는다.
* 고객사에서 사용하지 않는 모듈은 숨김을 기본으로 하되, 도입 안내가 필요한 경우 권한 있는 관리자에게만 범위를 설명한다.
* 제한 운영과 베타 기능은 운영중 기능과 동일한 무표시 상태로 섞지 않는다.
* 준비 중과 장기 검토 기능에 현재 건수, 완료 상태, 실행 CTA를 붙이지 않는다.
* 기능 상태가 변경되면 Feature Truth Inventory를 먼저 갱신하고 홈 노출을 조정한다.

## 7. Partner Demo와 정식앱 차이

역할 구조와 핵심 운영 흐름은 정합성을 유지하되 사용 목적과 데이터 의미는 구분한다.

| 항목 | Partner Demo | 정식앱 |
| --- | --- | --- |
| 목적 | 파트너 교육과 영업 시연 | 고객사의 실제 안전운영 기록 관리 |
| 모드 | 샘플 모드 | 실제 고객 운영 모드 |
| 데이터 | 샘플 데이터 | 고객사 실제 운영기록 |
| 저장 | 브라우저 안의 샘플 상태 | 고객사 운영 환경의 실제 저장 흐름 |
| 권한 | 역할별 흐름을 자유롭게 체험 | 고객사·사업장·역할 권한에 따라 제한 |
| 홈 우선순위 | 3분 내 핵심 흐름 체험 | 역할별 오늘 할 일 처리 |
| 기능 메뉴 | 체험에 필요한 샘플 기능으로 제한 | 사용 중인 전체 기능을 2차 메뉴로 제공 |
| 사진·음성 | 샘플 또는 시뮬레이션 가능 | 실제 운영 범위에서 기록 |
| 보고서 | 샘플 보고서 | 입력된 실제 운영기록 기반 요약 |
| 하단 안내 | 샘플 데이터와 실제 고객 환경 미연결 안내 | 현재 고객사 실제 운영 모드 안내 |

공통으로 유지할 원칙:

* 역할은 근로자, 현장관리자, 대표로 구분한다.
* TBM → 공유확인 → 위험제보 → 조치 → 현장비서 → 대표보고의 핵심 연결을 유지한다.
* 정식앱에 없는 기능을 Partner Demo에서 제공되는 것처럼 보이지 않는다.
* AI는 참고정보와 후보를 제안하며 최종 판단자가 아니다.

## 8. 금지 표현

다음 표현은 정식앱 홈의 카드, 배지, 빈 상태, 안내문, CTA, 보고서 진입 문구에 사용하지 않는다.

* `법적 면책`
* `처벌 방지`
* `완벽 대응`
* `자동 확정`
* `법적 증빙 완료`
* `안전조치 자동 완료`
* `위험성평가 대행`
* `KOSHA 인정 보장`
* `사고 자동 출동`
* `AI가 법적 판단 확정`

같은 의미를 우회해 약속하는 표현도 금지한다. 특히 확인, 접수, 저장, 사진첨부, 보고서 생성만으로 법적 적합성, 조치의 충분성, 안전상태 또는 책임 판단이 끝난 것처럼 표현하지 않는다.

권장 방향:

* `확인 기록을 남겼습니다.`
* `관리자 검토가 필요합니다.`
* `현재 입력된 운영기록 기준입니다.`
* `AI가 검토 후보를 제안했습니다.`
* `최종 판단과 책임은 관리자와 사업주에게 있습니다.`
