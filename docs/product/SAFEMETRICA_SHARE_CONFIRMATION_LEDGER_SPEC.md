# SafeMetrica Share Confirmation Ledger Spec v1

기준일: 2026-06-07

문서 상태: v1 제품·데이터 설계 기준

문서 범위: 공유확인 원장의 운영 기준과 향후 데이터 전환 기준을 정의하며, SQL·API·화면·DB 구현은 포함하지 않는다.

---

## 1. 한 줄 정의

공유확인 원장은 근로자가 위험요인, 위험성평가 공유내용, TBM, 현장 의견을 확인·제출한 기록을 회사·현장·날짜·근로자 기준으로 남기는 운영기록 원장이다.

---

## 2. 왜 필요한가

* 근로자 참여가 단순 화면 입력이나 일회성 알림에 머무르지 않고 실제 운영기록으로 남아야 한다.
* 현장관리자는 모든 기록을 일일이 읽는 대신 미확인자와 조치 필요 항목만 빠르게 확인해야 한다.
* 대표는 전체 입력내용이 아니라 미조치 현황, 반복 위험요인, 월간요약을 중심으로 확인해야 한다.
* 사고 발생 시 해당 회사·현장·기간의 관련 운영기록을 정리해 사후 설명자료로 활용할 수 있어야 한다.
* 1년치 운영기록을 일관된 기준으로 Export할 수 있어야 한다.

공유확인 원장은 근로자의 확인 또는 제출 사실과 그 이후의 관리자 검토·조치 흐름을 연결하기 위한 운영 기준이다. 확인기록 자체가 안전조치의 적정성, 법률상 의무 이행 또는 사고 책임에 관한 결론을 대신하지 않으며, 관리자 확인과 사업주 최종 판단이 별도로 필요하다.

---

## 3. 원장에 남길 최소 필드

아래 필드는 논리 데이터 모델 기준이다. 실제 저장소의 자료형, 필수 여부, 관계키, 인덱스 및 보존 정책은 구현 단계에서 별도로 확정한다.

| 필드 | 목적 | 기록 기준 |
| --- | --- | --- |
| `ledger_id` | 원장 레코드 식별자 | 각 확인·제출 이벤트를 고유하게 식별한다. |
| `company_code` | 회사 코드 | 회사별 조회·Export 범위를 구분한다. |
| `company_name` | 회사명 스냅샷 | 기록 발생 시점의 표시용 회사명을 남긴다. |
| `site_id` | 현장 식별자 | 현장별 기록과 예외 현황을 연결한다. |
| `site_name` | 현장명 스냅샷 | 기록 발생 시점의 표시용 현장명을 남긴다. |
| `work_date` | 작업일 | 실제 작업·TBM·공유확인의 기준 날짜를 남긴다. |
| `worker_id` | 근로자 식별자 | 동일 근로자의 확인 및 참여 기록을 연결한다. |
| `worker_name` | 근로자명 스냅샷 | 기록 발생 시점의 표시용 이름을 남긴다. |
| `worker_affiliation` | 소속 | 협력사, 팀 또는 공종 등 당시 소속을 남긴다. |
| `confirmation_type` | 확인·참여 유형 | 위험 확인, 공유확인, TBM 확인, 제보 등 이벤트 성격을 구분한다. |
| `source_module` | 발생 모듈 | 이벤트가 생성된 제품 모듈을 구분한다. |
| `risk_id` | 위험요인·위험성평가 참조 | 관련 기록이 없으면 비워 둘 수 있다. |
| `tbm_id` | TBM 참조 | 관련 기록이 없으면 비워 둘 수 있다. |
| `participation_id` | 현장참여 참조 | 관련 기록이 없으면 비워 둘 수 있다. |
| `confirmation_status` | 근로자 확인 상태 | 대기, 확인, 건너뜀, 확인 불필요를 구분한다. |
| `confirmed_at` | 확인 시각 | 근로자가 실제 확인한 시점을 남긴다. 미확인 상태에서는 비워 둘 수 있다. |
| `content_summary` | 확인 대상 요약 | 근로자가 확인한 위험요인·공유내용·조치내용의 요약 스냅샷을 남긴다. |
| `report_type` | 제보 유형 | 위험제보, 아차사고, 개선제안 등 제출 유형을 구분한다. |
| `report_title` | 제보 제목 | 제보 내용을 목록과 요약에서 식별할 수 있게 한다. |
| `report_content` | 제보 내용 | 근로자가 제출한 현장 의견 또는 사실 내용을 남긴다. |
| `location` | 발생 위치 | 현장 내 작업구역 또는 제보 위치를 남긴다. |
| `manager_review_status` | 관리자 검토 상태 | 접수부터 검토·조치판단·종결까지의 검토 흐름을 표시한다. |
| `manager_action_status` | 관리자 조치 상태 | 관리자 확인 및 실제 조치 진행 상태를 표시한다. |
| `manager_action_memo` | 관리자 조치 메모 | 조치 내용, 보류 사유, 추가 확인사항을 남긴다. |
| `evidence_item_ids` | 증빙 항목 참조 목록 | 사진 등 관련 증빙 항목을 연결한다. |
| `created_at` | 생성 시각 | 원장 레코드가 최초 생성된 시점을 남긴다. |
| `updated_at` | 수정 시각 | 상태·조치·연결 정보가 마지막으로 변경된 시점을 남긴다. |

### 3.1 필드 운영 원칙

* `company_name`, `site_name`, `worker_name`, `worker_affiliation`, `content_summary`는 당시 맥락을 보존하는 스냅샷 성격으로 관리한다.
* 식별자 필드와 표시용 스냅샷 필드를 함께 두어 이름이 변경되어도 과거 기록의 의미를 유지한다.
* `risk_id`, `tbm_id`, `participation_id`는 이벤트 성격에 따라 선택적으로 연결한다.
* 제보가 아닌 확인 이벤트에서는 `report_type`, `report_title`, `report_content`, `location`이 비어 있을 수 있다.
* 사진 등 증빙의 원본 자체가 아니라 증빙 항목 식별자를 `evidence_item_ids`로 연결한다.
* 상태 변경 시 누가 언제 무엇을 변경했는지 추적할 수 있는 별도 이력 구조를 향후 운영 DB에서 고려한다.
* 개인정보는 목적에 필요한 최소 범위로 다루며 실제 고객 개인정보나 운영 데이터를 설계 문서와 샘플에 넣지 않는다.

## 4. `confirmation_type` 정의

| 값 | 의미 | 대표 기록 |
| --- | --- | --- |
| `risk_check` | 오늘 위험요인 확인 | 작업일 기준으로 근로자가 당일 위험요인을 확인한 기록 |
| `risk_share_confirm` | 위험성평가 공유확인 | 위험성평가 내용과 안전조치 공유 여부를 확인한 기록 |
| `tbm_confirm` | TBM 확인 | 해당 작업일 또는 작업의 TBM 내용을 확인한 기록 |
| `worker_report` | 위험제보·아차사고·개선제안 | 근로자가 현장 의견이나 위험 관련 내용을 제출한 기록 |
| `action_acknowledge` | 조치내용 확인 | 접수된 의견 또는 위험요인에 대한 조치내용을 근로자가 확인한 기록 |
| `monthly_acknowledge` | 월간보고서 공유확인 후보 | 월간 안전운영 보고서의 공유확인으로 확장할 수 있는 후보 유형 |

`monthly_acknowledge`는 향후 확장을 위한 후보이며, 현재 운영 기능으로 오인되지 않도록 실제 제공 상태를 별도로 표시해야 한다.

---

## 5. `source_module` 정의

| 값 | 발생 기준 |
| --- | --- |
| `field_participation` | 근로자 현장참여 및 위험제보 흐름에서 생성 |
| `tbm` | TBM 조회·참석·확인 흐름에서 생성 |
| `risk_assessment` | 위험성평가 작성·공유·확인 흐름에서 생성 |
| `action` | 관리자 조치내용 공유 및 근로자 확인 흐름에서 생성 |
| `monthly_report` | 월간보고서 공유확인 흐름에서 생성하는 후보 |

`confirmation_type`은 기록의 행위를, `source_module`은 기록이 발생한 제품 흐름을 나타낸다. 두 값은 분리하여 동일 유형의 확인이 여러 모듈에서 발생하더라도 원인을 추적할 수 있게 한다.

## 6. 상태값 정의

### 6.1 `confirmation_status`

| 값 | 정의 |
| --- | --- |
| `pending` | 확인 요청 또는 확인 대상이 생성되었으나 근로자 확인이 완료되지 않은 상태 |
| `confirmed` | 근로자가 대상 내용을 확인한 시각이 기록된 상태 |
| `skipped` | 근로자가 해당 확인을 건너뛰었거나 운영상 미완료로 종료된 상태 |
| `not_required` | 근로자·작업·현장 조건상 해당 확인이 필요하지 않은 상태 |

`skipped`와 `not_required`는 동일하게 취급하지 않는다. `skipped`는 미확인 예외 검토 대상이 될 수 있고, `not_required`는 적용 제외 근거가 필요한 상태다.

### 6.2 `manager_review_status`

| 값 | 정의 |
| --- | --- |
| `접수` | 근로자 제보 또는 검토 대상 기록이 관리자 목록에 들어온 상태 |
| `검토중` | 관리자가 사실관계, 위험도 또는 대응 방향을 확인하는 상태 |
| `조치필요` | 현장 조치, 추가 확인 또는 책임자 배정이 필요하다고 판단한 상태 |
| `조치완료` | 관리자가 필요한 조치를 수행하고 검토 흐름을 완료로 표시한 상태 |
| `반려` | 중복, 대상 아님, 정보 부족 등의 사유와 함께 검토 대상에서 제외한 상태 |

### 6.3 `manager_action_status`

| 값 | 정의 |
| --- | --- |
| `미확인` | 관리자가 아직 조치 대상 여부를 확인하지 않은 상태 |
| `접수` | 관리자가 조치 대상을 확인하고 담당 흐름에 접수한 상태 |
| `조치중` | 현장 확인, 담당자 배정 또는 개선조치가 진행 중인 상태 |
| `조치완료` | 관리자가 조치 수행 결과를 기록한 상태 |
| `보류` | 즉시 조치하기 어렵거나 추가 판단이 필요해 사유와 함께 보류한 상태 |

관리자 상태값은 운영 진행상태다. 상태를 `조치완료`로 변경하더라도 조치의 충분성, 증빙의 적정성 및 최종 책임 판단은 관리자와 사업주가 별도로 확인해야 한다.

## 7. 근로자 화면 기준

근로자 홈에는 다음 4개 행동을 우선 노출한다.

1. 오늘 위험요인 확인
2. 위험성평가 공유확인
3. TBM 확인
4. 위험제보 / 아차사고 / 개선제안

### 7.1 노출 원칙

* 근로자에게 관리자·대표 기능을 과도하게 보여주지 않는다.
* 근로자는 확인·제보·의견 제출 중심의 짧고 명확한 흐름을 사용한다.
* 당일 확인 대상, 미완료 항목 및 제출 완료 여부를 우선 안내한다.
* 제출 후에는 접수 여부와 관리자 확인 필요 상태를 이해할 수 있게 표시한다.
* 다른 근로자의 개인정보, 관리자 내부 메모 및 대표용 경영 요약은 노출하지 않는다.
* 확인 행동과 제보 행동을 구분하여 제보 접수를 단순 확인 완료로 집계하지 않는다.

---

## 8. 현장관리자 화면 기준

현장관리자는 전체 원장 기록을 일일이 보는 방식이 아니라 다음 예외와 후속 행동 중심으로 본다.

* 미확인자
* 신규 제보
* 조치 필요 항목
* 조치사진 누락
* 반복 위험요인
* 월간보고서 반영 후보

### 8.1 요약 및 처리 원칙

* 회사·현장·작업일·상태 기준으로 예외 목록을 좁힐 수 있어야 한다.
* 미확인자는 `pending` 및 검토가 필요한 `skipped`를 중심으로 집계한다.
* 신규 제보는 접수 여부와 검토 대기시간을 구분해 보여준다.
* 조치 필요 항목은 담당, 현재 상태, 조치 메모 및 증빙 연결 여부를 함께 확인한다.
* 조치사진 누락은 사진이 항상 필수라는 의미가 아니라 해당 조치에 요구된 증빙이 연결되지 않은 예외를 뜻한다.
* 반복 위험요인은 동일·유사 위험의 반복 후보를 보여주되 최종 동일성 판단은 관리자가 수행한다.
* 월간보고서 반영 후보는 원장과 조치 이력을 바탕으로 선별하되 보고서 반영 여부는 관리자가 결정한다.

## 9. 대표 화면 기준

대표는 개별 입력 전문보다 다음 운영 요약을 중심으로 본다.

* 미조치 현황
* 반복 위험요인
* 위험제보 접수 현황
* 공유확인율 또는 미확인자 수
* 월간보고서
* 위험성평가표 출력지원

### 9.1 요약 원칙

* 대표 요약은 회사 전체에서 예외가 발생한 현장과 추세를 빠르게 파악하는 데 목적이 있다.
* 미조치와 장기 보류 항목은 현장·기간·위험유형 기준으로 내려볼 수 있어야 한다.
* 공유확인율을 제공할 때는 분모, 대상 기간, `not_required` 제외 기준을 함께 정의한다.
* 반복 위험요인은 확정 판단이 아니라 관리자 재검토가 필요한 후보로 표시한다.
* 월간보고서와 위험성평가표는 원장 기록을 연결해 출력 정리를 지원하되 관리자 확인과 사업주 최종 판단을 유지한다.
* 정식앱에는 실제 운영기록에서 계산되지 않은 가짜 숫자나 샘플 수치를 표시하지 않는다.

## 10. Supabase 전환 시 권장 테이블 후보

본 절은 장기 데이터 구조의 역할 후보만 정의한다. SQL, 스키마 생성, 접근정책 및 마이그레이션 구현은 이번 문서 범위에 포함하지 않는다.

| 테이블 후보 | 역할 |
| --- | --- |
| `share_confirmation_ledger` | 회사·현장·작업일·근로자 기준의 공유확인 원장 핵심 레코드 |
| `worker_profiles` | 근로자 식별, 소속 및 현장 배정의 기준 정보 |
| `worker_confirmation_events` | 확인 요청·확인·건너뜀 등 근로자 이벤트 상세 이력 |
| `action_review_logs` | 관리자 검토 및 조치 상태 변경 이력 |
| `evidence_items` | 사진 등 증빙 메타데이터와 원장·조치 연결 정보 |
| `monthly_report_items` | 월간보고서 반영 후보, 선택 결과 및 원장 참조 정보 |

권장 분리 원칙은 다음과 같다.

* 원장에는 조회와 Export에 필요한 핵심 현재 상태를 둔다.
* 상세 이벤트와 상태 변경은 별도 이력으로 보존하는 구조를 고려한다.
* 사진 파일과 업무 레코드는 분리하고 원장에서는 증빙 식별자로 연결한다.
* 회사 및 현장 범위가 모든 핵심 레코드에서 일관되게 유지되어야 한다.
* 보존기간, 접근권한, 삭제 및 정정 정책은 개인정보·노무·안전보건 운영 요구를 검토해 별도로 정한다.

## 11. Notion 운영과 Supabase 운영의 관계

* 현재 Notion은 백오피스·운영허브 역할을 한다.
* Supabase는 장기적으로 운영 SSOT와 핵심 트랜잭션 DB 역할을 한다.
* 전환 기간에는 Notion 저장과 Supabase shadow-write가 병행될 수 있다.
* 저장 대상이 병행되더라도 원장 필드, 상태값, 식별 기준 및 Export 의미는 동일해야 한다.
* 원장 기준은 향후 Supabase로 옮겨도 유지되어야 한다.
* 이중 저장 시 성공·실패·재시도·중복 방지·정합성 확인 기준은 구현 전에 별도 설계한다.
* Notion의 보기 구조나 수기 운영 편의가 원장의 핵심 의미를 변경해서는 안 된다.

## 12. Evidence Export 연결

Evidence Export는 1년치 운영기록 정리 또는 사고 발생 시 관련 기록의 사후 설명자료 구성을 지원하기 위한 추출 기준이다.

### 12.1 기본 필터

* 회사
* 현장
* 기간
* 작업일
* 근로자

### 12.2 포함 기록

* 위험요인
* TBM
* 공유확인
* 제보
* 조치
* 사진증빙
* 월간보고서

### 12.3 Export 원칙

* 회사와 현장 범위를 먼저 확정하고 시작일·종료일을 명시한다.
* 작업일과 시스템 생성·수정 시각을 구분해 제공한다.
* 근로자 확인기록과 관리자 검토·조치 이력을 시간 순서로 연결할 수 있어야 한다.
* 위험요인, TBM, 제보 및 조치의 참조 관계를 유지한다.
* 사진증빙은 연결 대상, 촬영·등록 시각, 설명 등 확인 가능한 메타정보와 함께 정리한다.
* 월간보고서 항목이 어떤 원장·조치 기록에서 반영되었는지 추적할 수 있어야 한다.
* Export 생성 시점, 적용 필터, 포함·제외 기준을 함께 남긴다.
* Export는 운영기록 정리 지원물이며 기록의 법률적 효력이나 조치의 충분성을 자체적으로 확정하지 않는다.

## 14. 데이터·보안 주의사항

* 실제 고객 개인정보를 본 문서, 예시 또는 테스트 자료에 넣지 않는다.
* 실제 고객 데이터나 인증·접속 비밀값을 문서에 기록하지 않는다.
* 샘플이 필요하면 식별 불가능한 가상 회사·현장·근로자 정보를 사용한다.
* 근로자 기록은 회사·현장·역할별 최소 접근 원칙을 적용할 수 있도록 설계한다.
* Export는 요청자 권한과 추출 범위를 확인하고, 불필요한 개인정보를 포함하지 않는 방향으로 설계한다.
* 보존, 정정, 삭제, 접근 이력에 관한 세부 정책은 실제 운영 및 관련 기준 검토 후 별도로 확정한다.

## 16. 고정 결론

SafeMetrica 공유확인 원장은 근로자의 확인·공유확인·제보, 관리자의 검토·조치, 대표의 예외·요약 확인을 하나의 추적 가능한 운영기록 기준으로 연결한다.

원장의 목적은 모든 입력을 관리자와 대표가 일일이 읽게 하는 것이 아니라, 미확인·미조치·반복 위험·증빙 누락 같은 예외를 빠르게 발견하고 후속 확인을 수행하게 하는 데 있다.

이 원장은 운영 확인기록 축적과 사후 설명자료 정리를 지원한다. 관리자 확인과 사업주 최종 판단을 대신하지 않으며 특정 법률상 결과, 처벌 회피 또는 사고 예방 결과를 보장하지 않는다.
