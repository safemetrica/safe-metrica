# SafeMetrica Manager Participation Review Status Spec v1

기준일: 2026-06-07

문서 상태: v1 제품·운영 분류 기준

문서 범위: 근로자 참여 기록을 관리자 화면에서 분류하고 우선순위를 정하는 기준을 정의한다. 화면·API·DB·Supabase·Notion·환경변수 구현과 데이터 마이그레이션은 포함하지 않는다.

---

## 1. 한 줄 정의

현장참여 검토 상태는 근로자의 공유확인·위험제보·아차사고·개선제안 기록을 현장관리자가 빠르게 확인하고 조치 여부를 판단하기 위한 운영 분류 기준이다.

---

## 2. 문서 목적

* 근로자 참여 기록을 관리자 화면에서 어떻게 분류해 보여줄지 기준을 정한다.
* 관리자 확인 시간을 줄이기 위해 전체 기록을 동일한 우선순위로 노출하지 않고 예외와 조치 필요 항목을 중심으로 정리한다.
* confirmation metadata v1의 `confirmation_type`, `confirmation_status`, `source_step`, `entry_intent`를 관리자 화면에서 어떻게 해석할지 정의한다.
* 향후 현장참여 접수함, 현장관리자 홈, 월간보고서 및 Evidence Export가 동일한 분류 의미를 사용하도록 연결 기준을 만든다.

---

## 3. 기본 원칙

* 근로자 기록을 모두 읽게 만들지 않는다.
* 관리자는 미확인, 신규 제보, 조치 필요, 조치 완료, 반복 위험을 우선 확인한다.
* 정상 공유확인은 자동 축적하고, 예외만 강조한다.
* AI는 분류 후보와 요약을 제안할 수 있지만 조치 확정자는 아니다.
* 최종 판단과 조치는 현장관리자와 사업주가 확인한다.
* 하나의 기록이 여러 분류 후보에 해당할 수 있다. 예를 들어 사진이 첨부된 위험제보는 `위험제보`, `조치 필요`, `월간보고서 반영 후보`에 동시에 포함될 수 있다.
* 자동 분류 결과는 원본 제출값을 변경하지 않는 표시·우선순위 기준으로 사용한다.
* 분류 근거가 부족하거나 값이 충돌하면 임의로 완료 처리하지 않고 `공유확인 미완료/확인 필요` 또는 `반려/보류` 등 관리자 확인이 필요한 상태로 남긴다.

---

## 4. 입력 데이터 기준

현재 또는 향후 다음 값을 관리자 분류에 사용할 수 있다. 값이 존재하지 않는 경우에는 추정값을 확정 상태처럼 표시하지 않는다.

| 입력값 | 관리자 분류에서의 용도 |
| --- | --- |
| `submission_type` | 제출 유형의 현재 표준값 또는 상위 구분값을 확인한다. |
| `legacy_type` | 과거 제출 유형을 보조적으로 해석한다. 표준값과 충돌하면 관리자 확인 대상으로 둔다. |
| `confirmation_type` | 공유확인인지 근로자 제보인지 등 참여 기록의 운영 성격을 구분한다. |
| `confirmation_status` | 공유확인 요청의 확인·대기 등 운영기록 상태를 구분한다. |
| `source_step` | 참여 흐름 중 어느 단계에서 기록이 제출되었는지 확인한다. |
| `entry_intent` | 사용자가 확인 또는 제보 중 어떤 의도로 진입했는지 우선순위 판단에 활용한다. |
| `title` | 관리자 목록의 대표 제목 및 요약 근거로 사용한다. |
| `content` | 위험, 아차사고, 개선 의견 및 조치 필요성의 검토 근거로 사용한다. |
| `location` | 현장 내 위치별 분류와 반복 위험 후보 판단에 활용한다. |
| `submitter` | 제출자 표시 및 후속 확인에 사용한다. |
| `anonymous` | 익명 제출 라벨과 개인정보 노출 범위를 결정한다. |
| `reported_date` | 접수일, 대기기간 및 월간보고서 대상 기간을 판단한다. |
| `status` | 기존 업무 상태가 있으면 조치 필요·완료·보류 해석의 보조값으로 사용한다. |
| `raw_payload.riskCheck` | 위험요인 확인 여부를 임시 분류의 보조 근거로 사용한다. |
| `raw_payload.riskAssessmentCheck` | 위험성평가 확인 여부를 임시 분류의 보조 근거로 사용한다. |
| `raw_payload.safetyMeasureCheck` | 안전조치 확인 여부를 임시 분류의 보조 근거로 사용한다. |
| `file_urls` | 사진 또는 첨부자료 존재 여부를 표시한다. 첨부 존재만으로 조치 완료로 판단하지 않는다. |
| `notion_page_id` | 현재 운영 백오피스 기록과의 연결 식별값으로 사용할 수 있다. |
| `notion_url` | 권한이 있는 관리자가 현재 운영 기록을 확인하는 연결값으로 사용할 수 있다. |

### 4.1 입력값 사용 원칙

* `confirmation_type`, `confirmation_status`, `source_step`, `entry_intent`를 우선 해석하고, 기존 필드와 `raw_payload`는 보조 근거로 사용한다.
* `submission_type`과 `legacy_type`은 현재·과거 데이터의 호환을 위한 값이며, 장기적으로 동일한 분류 의미로 정규화하는 것을 권장한다.
* `title`, `content`, `location`을 이용한 키워드 또는 AI 분류는 후보 생성에만 사용하며 관리자의 조치 판단을 대신하지 않는다.
* `anonymous=true`이면 목록에 익명 라벨을 표시하고, 불필요한 제출자 추정이나 개인정보 노출을 하지 않는다.
* `file_urls`가 하나 이상이면 사진 있음 라벨을 표시할 수 있으나 파일의 내용, 유효성 및 조치 증빙 적합성은 별도 확인한다.
* `notion_page_id`와 `notion_url`은 운영 기록 연결값이지 검토 또는 조치가 완료되었다는 근거가 아니다.

## 5. 관리자 분류 카테고리

아래 카테고리는 원본 제출 유형과 별도로 관리자 화면에서 사용할 운영 분류다. 복수 라벨을 허용하되, 목록의 대표 상태는 가장 높은 검토 우선순위를 따른다.

| 카테고리 | 분류 후보 기준 | 관리자 화면 처리 | 완료 또는 해제 기준 |
| --- | --- | --- | --- |
| 공유확인 완료 후보 | `confirmation_type=risk_share_confirm`이고 `confirmation_status=confirmed`인 기록 | 정상 축적하며 기본 목록에서는 낮은 우선순위로 표시한다. 필요할 때 회사·현장·날짜·근로자 기준으로 조회한다. | 운영상 확인 기록으로 유지한다. 이 상태만으로 조치나 법률상 판단이 끝난 것으로 보지 않는다. |
| 공유확인 미완료/확인 필요 | `confirmation_type=risk_share_confirm`이고 `confirmation_status=pending`이거나 확인값이 누락·충돌한 기록 | 확인 필요 예외로 강조하고 대상자·현장·작업일을 확인한다. | 유효한 확인 기록이 들어오거나 관리자가 확인 불필요·오류 등 사유를 검토해 별도 상태를 남긴다. |
| 위험제보 | `confirmation_type=worker_report` 또는 제출 유형·내용이 현장 위험 제보에 해당하는 기록 | 신규 제보로 우선 노출하고 위치, 내용, 사진, 익명 여부를 확인한다. | 관리자 검토 후 조치 필요, 조치 완료, 반려/보류 중 후속 상태를 연결한다. |
| 아차사고 | 제출 유형이나 내용이 사고로 이어질 뻔한 사건 보고에 해당하는 기록 | 발생일·위치·상황을 우선 확인하고 위험요인 및 재발 방지 검토로 연결한다. | 사실관계와 필요한 후속 조치를 관리자가 검토하고 상태 및 메모를 남긴다. |
| 개선제안 | 제출 유형이나 내용이 작업방법·시설·보호구·운영 개선 의견에 해당하는 기록 | 제안 내용과 적용 가능성을 검토하며 즉시 위험과 구분해 표시한다. | 채택·조치, 보류 또는 반려 판단과 사유를 남긴다. |
| 조치 필요 | 위험제보·아차사고·개선제안 중 추가 확인, 담당 지정, 현장 조치 또는 증빙 보완이 필요한 기록 | 높은 우선순위로 표시하고 담당, 기한, 조치 메모 및 필요한 사진 여부를 확인한다. | 관리자가 실제 조치 결과와 필요한 증빙을 확인해 조치 완료로 변경한다. |
| 조치 완료 | 관리자가 필요한 조치와 결과를 확인해 완료로 기록한 항목 | 완료일, 조치 내용 및 연결 증빙을 조회할 수 있게 유지한다. | 종결 후에도 원본 기록과 변경 이력을 유지하며, 재발·반복 위험이 확인되면 다시 검토할 수 있다. |
| 반려/보류 | 중복, 정보 부족, 대상 아님, 추가 확인 대기 또는 즉시 처리 곤란 등 사유가 있는 기록 | 사유와 재검토 조건을 함께 표시한다. 단순 숨김이나 삭제로 처리하지 않는다. | 정보 보완 후 재분류하거나, 관리자가 반려 사유를 남기고 종결한다. |
| 월간보고서 반영 후보 | 주요 위험제보, 아차사고, 개선·조치 사례, 반복 위험 또는 미조치 현황 등 월간 요약 가치가 있는 기록 | 보고서 후보 라벨을 표시하고 중복·민감정보·조치 상태를 검토한다. | 관리자가 실제 보고서 반영 여부와 요약 문구를 확인한다. 후보 상태만으로 자동 반영하지 않는다. |

### 5.1 대표 상태 우선순위

관리자 목록에서 하나의 대표 상태만 표시해야 할 때는 다음 순서를 권장한다.

1. 조치 필요
2. 공유확인 미완료/확인 필요
3. 위험제보 또는 아차사고
4. 반려/보류 중 재검토 기한이 도래한 항목
5. 개선제안
6. 조치 완료
7. 공유확인 완료 후보

`월간보고서 반영 후보`는 대표 처리 상태가 아니라 별도 보조 라벨로 표시한다. 반복 위험도 여러 기록을 묶어 재검토하게 하는 우선순위 신호이며, 개별 기록의 원본 유형을 대체하지 않는다.

## 6. confirmation metadata 해석 기준

| metadata 조건 | 관리자 분류 해석 | 표시 및 처리 원칙 |
| --- | --- | --- |
| `confirmation_type=risk_share_confirm`이고 `confirmation_status=confirmed` | 공유확인 완료 후보 | 정상 운영기록으로 축적하고 예외 목록에서는 낮은 우선순위로 둔다. |
| `confirmation_type=risk_share_confirm`이고 `confirmation_status=pending` | 공유확인 확인 필요 | 확인 필요 라벨을 표시하고 대상자·작업일·현장을 확인한다. |
| `confirmation_type=worker_report` | 관리자 검토 필요 | 신규 제보 후보로 우선 노출하고 세부 유형과 조치 필요성을 검토한다. |
| `entry_intent=report` | 제보 의도 진입이므로 우선 검토 후보 | 제출 유형이 불명확하더라도 일반 공유확인보다 먼저 검토한다. 실제 제보 유형은 내용과 원본값을 확인해 결정한다. |
| `source_step=3` | 의견/제보 단계에서 제출된 기록 | 위험제보·아차사고·개선제안 후보로 분류하되 `source_step`만으로 세부 유형이나 조치 상태를 확정하지 않는다. |

### 6.1 충돌 및 누락 처리

* `confirmation_type=risk_share_confirm`인데 `entry_intent=report`이거나 `source_step=3`이면 metadata 충돌 후보로 보고 관리자 확인 대상으로 둔다.
* `confirmation_status=confirmed`이더라도 `confirmation_type=worker_report`이면 제보에 대한 관리자 검토 필요 상태가 사라지지 않는다.
* `confirmation_type`이 없고 `source_step=3` 또는 `entry_intent=report`만 있으면 신규 제보 후보로 노출하되 세부 카테고리는 확정하지 않는다.
* metadata가 모두 없으면 `submission_type`, `legacy_type`, 기존 `status`, 내용 및 `raw_payload`를 이용해 임시 후보를 만들 수 있다. 근거가 부족하면 확인 필요로 둔다.
* `confirmed`는 운영기록 분류 상태이며 법률상 의무 이행, 안전조치의 적정성 또는 조치 완료를 의미하지 않는다.

## 7. 관리자 화면 표시 권장

현장참여 접수함의 카드 또는 리스트에는 긴 원문보다 검토 우선순위를 판단하는 데 필요한 최소 정보와 라벨을 먼저 표시한다.

| 권장 라벨 | 표시 조건 | 표시 목적 |
| --- | --- | --- |
| 공유확인 후보 | 공유확인 완료 후보에 해당하는 기록 | 정상 확인 기록과 제보 기록을 빠르게 구분한다. |
| 확인 필요 | 공유확인 대기, metadata 누락·충돌 또는 추가 확인이 필요한 기록 | 관리자가 확인해야 할 예외를 강조한다. |
| 신규 제보 | 아직 관리자 검토가 끝나지 않은 위험제보·아차사고·개선제안 후보 | 접수 후 최초 검토가 필요한 기록을 우선 노출한다. |
| 조치 필요 | 현장 조치, 담당 지정, 답변 또는 증빙 보완이 필요한 기록 | 후속 행동이 필요한 항목을 가장 높은 우선순위로 표시한다. |
| 사진 있음 | `file_urls`에 하나 이상의 첨부가 있는 기록 | 상세 화면을 열기 전에 첨부 존재 여부를 알린다. |
| 익명 | `anonymous=true`인 기록 | 제출자 표시 제한과 후속 소통 제약을 알린다. |
| 월간보고서 후보 | 월간 요약 가치가 있다고 선별된 기록 | 보고서 작성 시 재탐색할 항목을 모은다. |

### 7.1 카드·리스트 정보 순서

1. 대표 상태와 보조 라벨
2. 제목 또는 내용 요약
3. 현장·위치와 접수일
4. 제출 유형
5. 익명 여부와 사진 존재 여부
6. 담당·기한·조치 상태가 있는 경우 해당 정보
7. 원문 및 운영 백오피스 연결

AI 요약을 제공하는 경우에는 원문과 구분해 표시하고, 누락 가능성이 있음을 전제로 관리자가 원문을 확인할 수 있어야 한다.

---

## 8. 현장관리자 홈 카드 기준

현장관리자 홈의 오늘 할 일 4개는 현장참여 검토 상태와 다음처럼 연결한다.

| 홈 카드 | 연결 대상 | 권장 상태 문구 예시 |
| --- | --- | --- |
| 오늘 TBM 작성 | 당일 TBM 작성 또는 확인이 필요한 현장 업무 | `오늘 TBM 작성 필요` |
| 현장참여 접수함 | 신규 제보, 공유확인 확인 필요 및 미검토 참여 기록 | `신규 제보 확인 필요`, `공유확인 미완료 항목 확인` |
| 조치 필요 항목 | 조치 필요, 조치 증빙 보완 및 기한 도래 항목 | `조치사진 등록 필요` |
| 현장비서 요약 | 반복 위험, 주요 예외 및 월간보고서 반영 후보의 요약 | `월간보고서 반영 후보 확인` |

### 8.1 카드 집계 원칙

* 카드 숫자는 실제 연결 데이터에서 계산할 수 있을 때만 표시한다.
* 동일 기록이 여러 라벨을 가져도 카드 목적에 따라 중복 집계를 피할 수 있는 식별 기준을 둔다.
* `현장비서 요약`은 후보와 요약을 제공하며 조치 상태를 스스로 변경하지 않는다.
* 조치사진은 모든 항목에 일률적으로 요구하지 않고, 해당 조치에 사진 증빙이 필요하다고 정한 경우에만 누락 상태를 표시한다.
* 카드에서 항목을 열면 해당 분류의 근거가 되는 원본값과 제출내용을 확인할 수 있어야 한다.

### 8.2 금지 문구

다음 문구는 실제 운영기록의 범위를 넘어 완료, 법적 효력 또는 책임 결과를 단정하므로 사용하지 않는다.

* `전 직원 확인 완료`
* `법적 증빙 완료`
* `조치 자동 완료`
* `중대재해 면책`
* `과태료 방지`

대신 `대상 기록 기준 확인`, `공유확인 후보`, `관리자 검토 완료`, `조치 결과 확인`처럼 범위와 확인 주체를 드러내는 문구를 사용한다.

## 9. 대표 화면 연결 기준

대표에게는 전체 제출 목록이나 모든 원문을 기본 노출하지 않고 다음 요약을 전달한다.

* 미조치 항목 수
* 반복 위험요인
* 위험제보 접수 현황
* 공유확인율 또는 미확인자 수
* 월간보고서 반영 후보

### 9.1 요약 표시 원칙

* 미조치 항목은 대상 기간, 현장 및 조치 필요 상태의 집계 기준을 함께 정의한다.
* 반복 위험요인은 동일·유사 항목의 후보를 요약한 것이며 최종 동일성 및 위험성 판단은 관리자가 확인한다.
* 위험제보 접수 현황은 신규, 검토 중, 조치 필요, 조치 완료, 보류 등 운영 상태별로 구분할 수 있다.
* 공유확인율을 표시할 때는 분모, 대상 기간, 대상자 및 확인 불필요 항목의 제외 기준을 함께 제공한다.
* 미확인자 수를 표시할 때는 개인정보 접근권한과 업무상 필요한 범위를 적용한다.
* 월간보고서 반영 후보는 자동 보고 결과가 아니라 관리자 검토 대상을 뜻한다.
* 실제 데이터 연결 전에는 가짜 숫자, 임의의 비율 또는 샘플 집계를 표시하지 않는다. 데이터가 없으면 숫자 대신 연결 전 상태를 명확히 안내한다.

## 10. 월간보고서 및 Evidence Export 연결 기준

### 10.1 월간보고서

* 위험제보, 아차사고, 개선제안, 조치 필요·완료 및 반복 위험 후보를 월간보고서 반영 후보로 연결할 수 있다.
* 같은 사건에서 파생된 여러 기록은 식별 가능한 연결 기준을 사용해 중복 요약을 줄인다.
* 보고서에는 접수 현황, 주요 위험, 조치 경과 및 미조치 항목을 구분한다.
* 익명 제보와 개인정보는 보고 목적에 필요한 최소 범위로 정리한다.
* 후보 선택과 AI 요약은 초안 지원이며 최종 반영 문구는 관리자가 확인한다.

### 10.2 Evidence Export

* 회사, 현장, 기간, 작업일, 제출 유형, 검토 상태 및 조치 상태를 기준으로 관련 기록을 찾을 수 있어야 한다.
* 원본 제출값, confirmation metadata, 관리자 분류, 상태 변경, 조치 메모 및 첨부 연결을 구분해 내보내는 구조를 고려한다.
* 공유확인 완료 후보와 실제 관리자 조치 기록을 혼동하지 않도록 서로 다른 필드 또는 섹션으로 표시한다.
* Export는 운영기록 정리를 지원하는 기능이며 기록 자체의 적정성이나 법률상 결론을 자동으로 확정하지 않는다.
* 데이터 보존기간, 접근권한, 정정 이력 및 개인정보 처리 기준은 별도 정책과 구현 검토를 따른다.

## 11. 다음 코드 작업 제안

이번 문서에서는 구현하지 않으며, 후속 작업 후보로만 제안한다.

1. 현장참여 접수함에서 confirmation metadata 라벨 표시
   * `confirmation_type`, `confirmation_status`, `source_step`, `entry_intent`를 읽어 공유확인 후보, 확인 필요 및 신규 제보 라벨을 표시한다.
2. 관리자 홈 조치 필요 카드에 신규 제보/확인 필요 후보 연결
   * 실제 제출 데이터에서 계산되는 항목만 집계하고 상세 접수함으로 이동할 수 있게 한다.
3. `raw_payload` 기반 임시 분류 함수 작성
   * 표준 metadata가 없는 기존 기록에 한해 `riskCheck`, `riskAssessmentCheck`, `safetyMeasureCheck`를 보조 근거로 사용한다.
   * 임시 분류임을 구분하고 원본값을 변경하거나 완료 상태를 확정하지 않는다.
4. 추후 Supabase 원장 테이블 전환 시 동일 분류 기준 유지
   * 저장소가 바뀌어도 카테고리 의미, 우선순위, 충돌 처리 및 관리자 확인 원칙을 유지한다.

## 12. 비범위 및 주의사항

* 이 문서에는 실제 고객 개인정보를 넣지 않는다.
* 실제 인증정보, 비밀키 또는 환경설정 값을 넣지 않는다.
* SQL을 작성하지 않는다.
* DB migration을 수행하지 않는다.
* API, DB, Supabase, Notion 또는 환경변수를 수정하지 않는다.
* 이번 작업에서는 코드와 실행 동작을 수정하지 않는다.
* 확인, 검토, 조치 및 Export 상태를 법적 보장이나 책임 면제 표현으로 사용하지 않는다.
* 관리자가 분류 상태를 확인했다는 사실과 실제 현장 조치의 적정성은 구분해 기록한다.
